# 2. 선형구조 (Linear Data Structures)

## 2.1 선형 리스트와 연결 리스트의 비교

### (1) 선형 리스트 (Linear List)
* **정의**: 데이터를 논리적인 일렬 순서대로 연속적인 물리 메모리 공간에 배치하는 자료구조. 흔히 **배열 (Array)**로 구현됩니다.
* **이명**: 순차 리스트 (Ordered List), 연접 리스트 (Dense List).
* **조회 성능**: 인덱스를 알고 있다면 임의 접근(Random Access)이 가능하므로 $\mathcal{O}(1)$의 탐색 속도를 보장합니다.
* **삽입/삭제 연산의 대수적 오버헤드**:
  * **삽입 (Insert)**: 임의의 위치 $i$에 새 데이터를 삽입하려면, 인덱스 $i$부터 맨 끝까지의 모든 원소를 물리적으로 한 칸씩 뒤로 미는 작업을 동반해야 합니다.
    $$\text{평균 데이터 이동 횟수} = \frac{N+1}{2}$$
  * **삭제 (Delete)**: 임의의 위치 $i$의 원소를 삭제하려면, 그 빈틈을 메우기 위해 인덱스 $i+1$부터 끝까지의 모든 원소를 한 칸씩 앞으로 당겨야 합니다.
    $$\text{평균 데이터 이동 횟수} = \frac{N-1}{2}$$
  * *단점*: 삽입과 삭제가 빈번하게 발생하는 동적 시스템에서 대규모 데이터 이동 오버헤드로 인해 시스템 응답 성능에 심각한 저하가 야 발생합니다.

### (2) 연결 리스트 (Linked List)
* **정의**: 데이터 노드들이 물리적 메모리 상에 순차적이지 않고 흩어져 존재하며, 각 노드가 다음 노드의 메모리 주소를 포인터(링크)로 가리켜 논리적으로 체인 연결한 자료구조.
* **노드 (Node)의 구조**:
  ```
  ┌──────────────────┬──────────────────┐
  │  Data Field      │  Link Field      │ ───► Next Node Address
  └──────────────────┴──────────────────┘
  ```
* **탐색 성능**: 특정 노드를 찾기 위해 첫 번째 시작 노드인 `head`에서부터 링크 포인터를 순차적으로 타고 가야 하므로 임의 접근이 불가하며 $\mathcal{O}(n)$의 시간이 소요됩니다.
* **삽입/삭제 연산 성능**: 물리적 데이터의 이동이 전혀 없으며 오직 포인터 값의 대입 갱신 연산($2$회)만 필요하므로 $\mathcal{O}(1)$의 우수한 시간 성능을 나타냅니다.
* **형태 종류**:
  * **단일 연결 리스트 (Singly Linked List)**: 링크 필드가 다음 노드만을 단방향으로 지시.
  * **원형 연결 리스트 (Circular Linked List)**: 마지막 노드의 링크가 `Null` 대신 다시 첫 번째 노드(`head`)를 가리켜 순환 루프를 형성하는 구조.
  * **이중 연결 리스트 (Doubly Linked List)**: 각 노드가 이전 노드를 가리키는 `prev` 링크와 다음 노드를 가리키는 `next` 링크 2개를 가집니다. 역방향 탐색 및 손상된 링크 복구가 가능하나, 포인터 메모리 소요가 더 큽니다.
  * **이중 원형 연결 리스트 (Doubly Circular Linked List)**: 이중 연결 리스트의 물리적 첫 노드와 마지막 노드를 상호 연결하여 환형 구조로 회전시킨 구조.

## 2.2 스택 (Stack)

### (1) 동작 원리와 핵심 개념
* **원리**: **LIFO (Last-In First-Out, 후입선출)**. 가장 나중에 밀어 넣은 데이터가 구조 상 가장 먼저 제거되어 출력되는 제한적인 선형 자료구조.
* **주요 용어**:
  * `push(item)`: 스택의 가장 높은 상단부 위치인 `top`에 새로운 원소를 적재함.
  * `pop()`: 현재 `top` 위치에 적재되어 있는 원소를 구조에서 꺼내어 제거하고 동시에 반환함.
  * `top`: 스택 최상단 데이터가 들어 있는 실제 인덱스를 지시하는 제어 포인터.
  * `peek()`: `top` 데이터를 제거하지 않고 어떤 값인지 단순 조회만 수행함.

### (2) 하드웨어적 오류 상황
* **스택 오버플로 (Stack Overflow)**: 이미 설정된 스택의 최대 허용 한계 용량(`capacity`)이 가득 찬 포화 상태인데도 추가로 `push`를 시도하여 주소 범위를 이탈할 때 발생하는 오류.
* **스택 언더플로 (Stack Underflow)**: 스택에 저장된 데이터가 전혀 없는 공백 상태(`top == -1`)임에도 불구하고 `pop` 또는 `peek` 명령을 가동하여 존재하지 않는 쓰레기 메모리 값을 취하려 할 때 발생하는 오류.

### (3) 정밀 추적 퀴즈 풀이 과정

#### [퀴즈 1] A, B, C, D의 입력 흐름에서 "B -> C -> D -> A" 출력을 얻기 위한 스택 제어 순서
* **목표 출력**: B, C, D, A 순서로 pop이 수행되어야 합니다.
* **수학적 추적 단계**:
  1. `push(A)`: 스택 상태 `[A]` (A가 바닥에 놓임)
  2. `push(B)`: 스택 상태 `[A, B]`
  3. `pop()` 실행 $\rightarrow$ **B가 가장 먼저 탈출하여 출력됨** (스택 상태 `[A]`)
  4. `push(C)`: 스택 상태 `[A, C]`
  5. `pop()` 실행 $\rightarrow$ **C가 탈출하여 출력됨** (스택 상태 `[A]`)
  6. `push(D)`: 스택 상태 `[A, D]`
  7. `pop()` 실행 $\rightarrow$ **D가 탈출하여 출력됨** (스택 상태 `[A]`)
  8. `pop()` 실행 $\rightarrow$ **가장 바닥에 남았던 A가 최종 탈출하여 출력됨** (스택 상태 `[]`)
* **최종 연산 시퀀스**: `push(A) -> push(B) -> pop() -> push(C) -> pop() -> push(D) -> pop() -> pop()`

#### [퀴즈 2] 아래 Python 반복문 코드 실행 후 스택 `value`에 남는 최종 요소 추적
```python
value = []
for i in range(20):
    if i % 3 == 0:
        value.append(i)
    elif i % 4 == 0:
        if len(value) > 0:
            value.pop()
```
* **컴퓨터 로직 조건 분석**:
  * $i$가 $3$의 배수이면 스택에 `append` (push) 수행.
  * $i$가 $3$의 배수가 아니고 $4$의 배수이며 스택이 비어있지 않다면 `pop` 수행.
* **루프 전체 추적 테이블 ($i = 0 \sim 19$)**:
  * `i = 0`: $3$의 배수 $\rightarrow$ `append(0)` $\rightarrow$ `value = [0]`
  * `i = 1`, `i = 2`: 조건 없음 $\rightarrow$ 변화 없음
  * `i = 3`: $3$의 배수 $\rightarrow$ `append(3)` $\rightarrow$ `value = [0, 3]`
  * `i = 4`: $4$의 배수 $\rightarrow$ `pop()` $\rightarrow$ `value = [0]`
  * `i = 5`: 조건 없음 $\rightarrow$ 변화 없음
  * `i = 6`: $3$의 배수 $\rightarrow$ `append(6)` $\rightarrow$ `value = [0, 6]`
  * `i = 7`: 조건 없음 $\rightarrow$ 변화 없음
  * `i = 8`: $4$의 배수 $\rightarrow$ `pop()` $\rightarrow$ `value = [0]`
  * `i = 9`: $3$의 배수 $\rightarrow$ `append(9)` $\rightarrow$ `value = [0, 9]`
  * `i = 10`, `i = 11`: 조건 없음 $\rightarrow$ 변화 없음
  * `i = 12`: $3$의 배수 ($12 \pmod 3 == 0$이므로 첫 `if` 분기 진입) $\rightarrow$ `append(12)` $\rightarrow$ `value = [0, 9, 12]`
  * `i = 13`, `i = 14`: 조건 없음 $\rightarrow$ 변화 없음
  * `i = 15`: $3$의 배수 $\rightarrow$ `append(15)` $\rightarrow$ `value = [0, 9, 12, 15]`
  * `i = 16`: $4$의 배수 $\rightarrow$ `pop()` $\rightarrow$ `value = [0, 9, 12]`
  * `i = 17`: 조건 없음 $\rightarrow$ 변화 없음
  * `i = 18`: $3$의 배수 $\rightarrow$ `append(18)` $\rightarrow$ `value = [0, 9, 12, 18]`
  * `i = 19`: 조건 없음 $\rightarrow$ 변화 없음
* **최종 결과**: `value`에 순서대로 `[0, 9, 12, 18]`이 적재되어 잔존함.

## 2.3 큐 (Queue)

### (1) 동작 원리와 선형 큐의 단점
* **원리**: **FIFO (First-In First-Out, 선입선출)**. 먼저 유입된 데이터가 대기 행렬 순서에 따라 가장 먼저 빠져나가는 구조. 삽입은 `rear` 포인터에서, 삭제는 `front` 포인터에서 각각 분리 수행됩니다.
* **선형 큐 (Linear Queue)의 태생적 한계**:
  데이터의 입력(`enqueue`)과 출력(`dequeue`)이 거듭될수록 `rear`와 `front` 인덱스 값은 우측 배열 끝자락을 향해 지속해서 증가합니다. 이로 인해 앞쪽의 삭제 완료 공간이 완전히 비어있음에도 불구하고 `rear`가 물리 배열의 마지막 인덱스에 도달하게 되면 포화 상태(`rear == capacity - 1`)로 오인하여 추가 데이터 적재를 차단하는 오동작이 야기됩니다.

### (2) 원형 큐 (Circular Queue)의 구현 기하학
배열의 끝이 물리적으로 다시 처음 인덱스와 둥글게 결합해 도는 것처럼 인덱스 모듈러(Remainder) 연산을 사용하는 원형 회전 큐를 설계하여 선형 큐의 낭비를 원천 차단합니다.
* **인덱스 회전 공식**: 
  $$\text{Next Index} = (\text{Current Index} + 1) \pmod{\text{capacity}}$$

#### 공백(Empty)과 포화(Full) 상태의 오판 방지 기법
원형 공간에서 데이터가 전혀 들어있지 않은 초기 공백 조건도 `front == rear`가 성립하며, 모든 공간에 데이터를 꽉 채우는 완전 포화 조건 또한 `rear`가 회전하여 `front`와 같은 위치를 가리키게 되어 `front == rear`가 됩니다.
이 모호성을 차단하기 위해 **"한 칸을 항상 빈 여백으로 남겨두는 오프셋 전략"**을 기본 법칙으로 설계합니다.

```
       [원형 큐의 포화 상태 기하]
       
             (front) 0
             ┌───────┐
           7 │  Empty│ 1
             ├───────┼───────
        rear │  Data │ Data
           6 │ (Full)│ 2
             ├───────┼───────
             │  Data │ Data
           5 └───────┘ 3
                 4
       
       * 포화 조건: (rear + 1) % 8 == front
```

* **초기/공백 상태 제어 포인터 지정**:
  $$\text{front} = 0, \quad \text{rear} = 0$$
* **공백 판정식 (Empty Condition)**:
  $$\text{front} == \text{rear}$$
* **포화 판정식 (Full Condition)**:
  $$(\text{rear} + 1) \pmod{\text{capacity}} == \text{front}$$

## 2.4 데크 (Deque)와 변형 데크

데크(Double Ended Queue)는 선형 자료구조의 양단(Front 및 Rear) 모든 위치에서 삽입과 삭제를 자유롭게 허용하는 하이브리드 입출력 구조입니다.
* **입력제한 데크 (Scroll)**: 데이터 삽입은 오직 리스트의 한쪽 단부로만 가능하며, 삭제는 양쪽 끝단 모두에서 허용하는 특화 구조.
* **출력제한 데크 (Shelf)**: 데이터 삽입은 양쪽 끝단 모두에서 자유롭게 허용하나, 삭제 출력은 오직 정해진 한쪽 단부에서만 가능하게 제한한 구조.
