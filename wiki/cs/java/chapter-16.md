# Summary
MVC(Model-View-Controller) 패턴은 애플리케이션의 역할을 3가지의 독립된 영역(데이터 모델, 사용자 인터페이스, 이벤트 제어 로직)으로 격리하는 소프트웨어 설계 패턴입니다. 사용자 인터페이스(UI) 개발의 복잡도를 낮추고 유지보수성을 향상시키는 "관심사의 분리(Separation of Concerns)" 실현을 궁극적인 목적으로 삼으며, 본 문서에서는 각 컴포넌트의 유기적 상호작용과 데이터 플로우 메커니즘을 상세히 규명합니다.

# Why it matters
- **관심사의 분리 (Separation of Concerns)**: 비즈니스 도메인 규칙을 다루는 코드와 화면 UI를 렌더링하는 코드가 뒤엉키게 되면, 작은 UI 디자인 변경 요청에도 도메인 로직 전체를 다시 컴파일하고 테스트해야 하는 병폐가 발생합니다. MVC 패턴은 이를 엄격히 구분하여 코드 결합도(Coupling)를 낮추고 응집도(Cohesion)를 극대화합니다.
- **유지보수와 확장성**: 뷰(View)와 모델(Model)을 독립시킴으로써, 백엔드 로직을 전혀 변경하지 않은 채 데스크톱 응용 프로그램(Java Swing) 환경에서 웹 브라우저(HTML/JS) 혹은 모바일 앱 인터페이스로 화면 구성을 쉽게 전이할 수 있습니다.

# Key Ideas

1. **MVC 구성 요소와 독립된 역할**:
   - **모델 (Model)**:
     - 애플리케이션의 핵심 데이터 상태와 비즈니스 로직(데이터 가공, 연산 등)을 전담합니다.
     - **POJO (Plain Old Java Object)** 형태의 순수 데이터 객체(Entity, DO, DTO)와 데이터베이스 연동을 직접 대행하는 **DAO (Data Access Object)** 등이 여기에 속합니다.
     - 뷰(View)나 컨트롤러(Controller)의 인스턴스를 소유하거나 이들의 라이브러리에 종속되지 않는 **순수 자바 코드로 작성**하는 것이 원칙입니다.
   - **뷰 (View)**:
     - 화면 구성과 출력을 담당하는 시각적 UI 컴포넌트들입니다 (Java Swing의 `JFrame`, `JPanel`, `JButton` 등).
     - 주어진 데이터를 받아 사용자에게 보여주는 용도로만 사용하는 것이 이상적입니다.
     - **뷰가 데이터베이스나 비즈니스 로직을 직접 실행하여 데이터를 불러오는 구조는 안티패턴(Anti-Pattern)**으로 배제되어야 합니다.
     - 모델 및 컨트롤러와의 결합성을 최소화하여 뷰 단독으로 교환 가능하게 설계해야 합니다.
   - **컨트롤러 (Controller)**:
     - 사용자로부터 들어오는 요청(Request)과 이벤트(Event)를 포착하고 처리하는 두뇌 역할을 합니다.
     - Java Swing에서는 버튼 클릭이나 키 입력을 수신하는 `ActionListener` 인터페이스 구현체가 컨트롤러의 역할을 담당합니다.
     - 사용자 입력을 기반으로 모델(Model)의 상태를 수정하거나 비즈니스 메소드를 호출하고, 뷰(View)를 새로고침(Refresh)하도록 지시합니다.
     - 요청 처리를 진행하기 위해 특정 모델과 뷰를 인스턴스로 바인딩해야 하므로, **뷰와 불가피하게 일부 종속관계(Dependency)가 형성**됩니다.

2. **MVC 데이터 흐름 (Event Flow)**:
   - 사용자와 시스템 간의 유기적 통신은 아래와 같은 엄격한 순차 순환 고리를 형성합니다.
   ```
   [사용자 액션 (UI 조작)] 
           │
           ▼
   [컨트롤러 감지 (Event Listener)]
           │
           ▼
   [모델 업데이트 (Data/DAO 처리)]
           │
           ▼
   [상태 변화 알림 (Model -> View / Observer Pattern)]
           │
           ▼
   [뷰 재렌더링 (화면 출력 및 데이터 갱신)]
   ```
   1. **사용자 액션**: 사용자가 화면(View) 상의 버튼을 클릭하거나 텍스트 필드에 데이터를 입력합니다.
   2. **컨트롤러 감지**: 뷰 컴포넌트에 할당된 컨트롤러(Event Listener)가 해당 이벤트를 포착합니다.
   3. **모델 업데이트**: 컨트롤러가 전송된 입력을 처리하기 위해 모델(Model)의 도메인 로직이나 DAO의 CRUD 연산을 실행하여 상태를 갱신합니다.
   4. **상태 변화 알림**: 모델의 데이터가 성공적으로 갱신되었음을 알립니다. 필요에 따라 옵저버(Observer) 디자인 패턴을 활용하여 모델이 직접 등록된 뷰들에게 알리거나, 컨트롤러가 중재하여 알림을 전달합니다.
   5. **뷰 재렌더링**: 변경된 모델 데이터를 읽어와 사용자 화면을 다시 그리고 가시적으로 최신 데이터를 반영합니다.

3. **버튼 이벤트 처리 구조 (Button Event Handling Structure)**:
   - 실제 Swing 기반 코드에서 회원 수정 기능을 예시로 든 흐름도:
   ```
   [View] (수정 버튼 클릭) ──▶ [Controller: updateListener] ──▶ [Model/DAO: updateMember()]
                                         │
                                         ▼
                                  [View: 화면 갱신] ◀── (SELECT 결과 기반 재렌더링)
   ```
   - 뷰 컴포넌트(버튼)에 바인딩된 `updateListener`가 트리거되면, 뷰의 입력 필드 정보를 수집하여 컨트롤러가 모델 객체를 채운 후, `DAO`의 `updateMember(member)`를 호출해 물리적 영속성 레이어(RDBMS)에 저장합니다. 이후 성공 알림 및 리스트 갱신 메소드를 통해 뷰를 리렌더링합니다.

4. **MVC 패턴의 핵심 장점 및 효용성**:
   - **병렬 개발 가속화**: 화면단 설계 팀과 백엔드 데이터베이스 비즈니스 로직 구현 팀이 인터페이스 규약만 맞추면 개발 완료를 대기할 필요 없이 동시에 독립적인 개발을 완수할 수 있습니다.
   - **코드의 재사용성**: 데이터 처리에 대한 모델 규칙이 순수 자바로 격리되어 있으므로, 동일한 모델(Model) 로직을 손대지 않고 콘솔 인터페이스, GUI 스윙 인터페이스, Web API 서버 인터페이스 등으로 화면 채널을 다각화할 수 있습니다.
   - **유지보수 비용 절감**: 사용자 요구 사항에 따른 잦은 레이아웃 변경(HTML 태그나 Swing 컴포넌트 변경)이 핵심 데이터 구조 및 RDBMS 로직에 어떠한 충격도 가하지 않습니다.
   - **테스트 용이성**: 복잡한 GUI 화면 구동 시뮬레이션 없이도, 비즈니스 도메인을 담당하는 핵심 모델(Model) 클래스만을 떼어내어 JUnit 등을 통한 **단위 테스트(Unit Test)**를 완벽하게 검증해 낼 수 있어 코드 신뢰성이 높아집니다.

# Related Concepts
- **MVP (Model-View-Presenter)**: MVC 패턴에서 파생된 형태로, Presenter가 View와 Model 사이에서 완전한 중재자가 되어 View와 Model 간의 의존성을 100% 제거한 패턴입니다.
- **MVVM (Model-View-ViewModel)**: 데이터 바인딩(Data Binding) 기술을 활용하여 View의 상태 변화와 ViewModel의 상태 변화를 선언적으로 연동함으로써 컨트롤러의 중재 코드 분량을 대폭 감소시킨 현대적 UI 패턴입니다.
