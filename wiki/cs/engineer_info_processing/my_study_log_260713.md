## 66. C언어 - 포인터 매개변수의 독립성(Call by Value) 및 문자열 출력(%s) 메커니즘

### 문제
아래 C 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 1라인의 `include〈stdio.h〉` 및 8라인의 `char *p=a；` 오타는 정상적인 문법으로 보정하여 풀이한다.)
```c
#include <stdio.h>
char* fn(char *p){
    p++;
    return p;
}
int main(){
    char a[]= "abcd";
    char *p = a; 
    printf("%s\n", fn(p));
    printf("%s\n", p);
    return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `b(\n)bcd` (포인터 연산 메커니즘은 유추했으나, %s 서식자의 널 문자 종결 탐색 특성 및 매개변수 값 복사 독립성 혼동으로 오답)
* **모범 답안**:
  ```text
  bcd
  abcd
  ```
* **결과**: **오답 분석 및 이론 완벽 교정**

### 핵심 해설
이 문제는 C언어 시험에서 수험생들을 낚기 위해 매번 즐겨 출제하는 두 가지의 핵심 개념 함정이 결합되어 있습니다.

* **함정 1: `%s` (문자열) 출력 포맷의 동작 방식**
  * `%c`는 가리키는 주소의 단 한 글자(Character)만 출력합니다.
  * 반면 **`%s`**는 전달받은 주소(Pointer)를 시작점으로 삼아, 메모리를 1바이트씩 계속 이동하면서 **널 문자(`\0`)를 만날 때까지의 모든 문자를 하나의 문자열로 묶어 출력**합니다.
  * `fn(p)`가 반환한 것은 `'b'`의 주소이므로, 그 주소부터 널 문자 전까지 출력하게 되어 `"b"`가 아닌 **`"bcd"`**가 출력됩니다.
* **함정 2: 포인터 매개변수의 독립성 (Call by Value)**
  * `main`에서 `fn(p)`를 호출할 때 전달하는 `p`는 포인터 변수 `p`에 담긴 **주소값**입니다.
  * 함수 `fn(char *p)`의 매개변수 `p`는 `main`의 `p`와 이름만 같을 뿐, 메모리 영역이 독립된 별개의 임시 포인터 변수입니다 (주소값만 복사되어 대입됨).
  * 따라서 함수 내부에서 `p++`를 백날 수행하더라도, `main` 함수 영역에 위치한 원본 포인터 변수 `p`에는 **아무런 영향이 없습니다.**
  * 즉, `main`의 포인터 `p`는 여전히 문자열의 시작인 `'a'`의 주소를 가리키고 있습니다.
  * 이에 따라 두 번째 출력문 `printf("%s\n", p);`는 원래 문자열인 **`"abcd"`**를 온전히 출력합니다.

* **💡 사용자 메타인지 피드백 분석 (핵심 오답 요인)**:
  * 본 문제 풀이 시 무의식적으로 **`p = fn(p);`** 와 같은 반환값 대입 연산이 일어났다고 가정하고 연산 흐름을 추적하여 오답이 도출되었습니다.
  * 만약 코드에 대입 연산인 `p = fn(p);` 가 포함되어 있었다면, `main`의 원본 포인터 `p`가 `'b'`의 주소값으로 업데이트되어 두 출력문 모두 `bcd`를 출력하게 됩니다.
  * 그러나 대입문이 없고 단순히 반환된 임시 주소만 1회성 출력(`printf("%s\n", fn(p));`)에 썼기 때문에, 원본 `p`는 여전히 `'a'`의 주소를 유지하여 두 번째 출력에서 `abcd`가 나오게 된 것입니다. 이 **대입 연산자의 유무**를 구분하는 것이 득점을 결정짓는 핵심 포인트입니다.

## 67. C언어 - 제어문 (switch-case 제어 구조에서의 break 누락 및 Fall-through 현상)

### 문제
아래 C 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 1라인의 `#include <stdio.h)` 및 10라인의 `surmb=i；` 등 문장 부호 OCR 오타는 각각 `#include <stdio.h>`와 `sum+=i;` 문맥으로 보정하여 풀이한다.)
```c
#include <stdio.h>
int main(){
    int i=2;
    int sum=0;
    for(;i<4;){
        switch(i%2){
            case 0 :
                sum+=i;
            default:
                sum+=i;
        }
        i+=3;
    }
    printf("%d", sum);
    return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `4` (break 생략에 따른 switch문 하향 순차 처리 연산 완벽 적용)
* **모범 답안**: **`4`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **특이한 `for`문 구조 분석**:
  * `for(; i < 4; )` 처럼 초기화식과 증감식이 생략되어 있습니다. 이는 `while(i < 4)`와 완전히 동일한 조건 제어 흐름을 의미합니다.
* **루프 1회전 (i=2)**:
  * 조건 검사: `i < 4` $\rightarrow$ `2 < 4` (참, 진입)
  * **`switch(i % 2)` 분기**:
    * `2 % 2 = 0` 이므로 **`case 0:`**으로 분기 및 진입합니다.
    * `sum += i;` 가 수행되어 `sum = 0 + 2 = 2`가 됩니다.
    * **핵심 함정**: `case 0:` 블록 끝에 제어 제어 명령인 **`break;`**가 존재하지 않습니다!
    * **Fall-through (흘러내림)**: `break;`가 없으므로 제어가 밑으로 계속 흘러내려 아래에 있는 **`default:`** 블록의 `sum += i;`까지 연속해서 연쇄 실행됩니다.
    * 이에 따라 `sum`에 `i` 값인 `2`가 한 번 더 누적되어 `sum = 2 + 2 = 4`가 됩니다.
  * **증감식 실행**: `i += 3;` 에 의해 `i`는 `2 + 3 = 5`가 됩니다.
* **루프 2회전 (i=5)**:
  * 조건 검사: `i < 4` $\rightarrow$ `5 < 4` (거짓, 루프 종료)
* **출력**:
  * 최종적으로 누적된 `sum` 변수의 값인 **`4`**가 화면에 정상 출력됩니다.

## 68. 소프트웨어 설계 - 모듈 결합도(Coupling) 유형 매칭

### 문제
다음은 결합도의 유형에 대한 설명이다. 괄호 ( ) 안에 알맞은 결합도를 보기에서 찾아 쓰시오.
* ( ① )는 파라미터가 아닌 모듈 밖에 선언되어 있는 전역 변수를 참조하고 전역 변수를 갱신하는 식으로 상호 작용하는 경우의 결합도이다.
* ( ② )는 모듈 간의 인터페이스로 배열이나 객체, 구조 등이 전달되는 경우의 결합도이다.
* ( ③ )는 두 개의 모듈이 외부에서 도입된 데이터 포맷, 통신 프로토콜, 또는 디바이스 인터페이스를 공유할 경우의 결합도이다.

[보기]
(ㄱ) Content Coupling (ㄴ) Common Coupling (ㄷ) External Coupling (ㄹ) Stamp Coupling (ㅁ) Control Coupling (ㅂ) Data Coupling

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: ㄴ. Common Coupling (공통 결합도)**
  * **②: ㄹ. Stamp Coupling (스탬프 결합도)**
  * **③: ㄷ. External Coupling (외부 결합도)**
* **모범 답안**:
  * **①: Common Coupling (또는 공통 결합도, ㄴ)**
  * **②: Stamp Coupling (또는 스탬프 결합도, ㄹ)**
  * **③: External Coupling (또는 외부 결합도, ㄷ)**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
모듈 간의 상호 의존성을 나타내는 결합도(Coupling)는 낮을수록 우수한 설계이며, 총 6가지 유형이 존재합니다.

* **지문 핵심 단서 및 결합도 분석**:
  1. **공통 결합도 (Common Coupling) (지문 ①번)**:
     * *단서*: **`전역 변수`**, **`공동 자료 영역`**을 여러 모듈이 공유 및 갱신함.
  2. **스탬프 결합도 (Stamp Coupling) (지문 ②번)**:
     * *단서*: 인터페이스로 단순 변수가 아닌 **`배열`**, **`객체`**, **`구조체`** 등 동일한 자료 구조를 전달함.
  3. **외부 결합도 (External Coupling) (지문 ③번)**:
     * *단서*: **`외부 프로토콜`**, **`외부 데이터 포맷`**, **`디바이스 인터페이스`**를 공유함.
* **결합도 강도 순서 (약함 $\rightarrow$ 강함 / 좋은 설계 $\rightarrow$ 나쁜 설계)**:
  * **자 $\rightarrow$ 스 $\rightarrow$ 제 $\rightarrow$ 외 $\rightarrow$ 공 $\rightarrow$ 내**
    * **자**료 결합도 (Data) $\rightarrow$ **스**탬프 결합도 (Stamp) $\rightarrow$ **제**어 결합도 (Control) $\rightarrow$ **외**부 결합도 (External) $\rightarrow$ **공**통 결합도 (Common) $\rightarrow$ **내**용 결합도 (Content)

## 69. 데이터베이스 - 병행 제어(Concurrency Control) 및 분산 트랜잭션 보장 기법

### 문제
다음은 병행 제어 기법에 대한 설명이다. 괄호 ( ) 안에 알맞은 기법을 쓰시오.
* ( ① )은/는 트랜잭션이 어떠한 검증도 수행하지 않고 일단 트랜잭션을 수행하고, 트랜잭션 종료 시 검증을 수행하여 데이터베이스에 반영하는 기법이다.
* ( ② )은/는 여러 개의 분산 데이터베이스 시스템에서 트랜잭션의 일관성을 유지하기 위한 기법으로 트랜잭션을 두 단계로 분리하여 제어하는 기법이다.

### 정답 및 풀이 결과
* **작성한 답**: 미작성 (생소한 DB 동시성 제어 개념으로 에이저트 정밀 이론 해설 분석 요청)
* **모범 답안**:
  * **①: 낙관적 검증 기법 (또는 낙관적 기법, 낙관적 병행 제어, Optimistic Concurrency Control)**
  * **②: 2단계 커밋 (또는 2-Phase Commit, 2PC)**
* **결과**: **이론 완벽 정리 완료**

### 핵심 해설
이 문제는 단일 DB의 동시성을 보장하는 **병행 제어(Concurrency Control)** 핵심 기법과, 분산 환경에서의 합의 알고리즘인 **분산 트랜잭션 제어 프로토콜**을 연동한 중상 난이도의 문제입니다.

* **지문 핵심 단어 및 분석**:
  1. **낙관적 검증 기법 (Optimistic Concurrency Control) (지문 ①번)**:
     * *개념*: 트랜잭션 실행 동안에는 충돌이 거의 일어나지 않을 것이라 '낙관적'으로 가정하고 **검증 없이 일단 수행**합니다.
     * *작동 단계*: 트랜잭션을 수행하는 **판독(Read) 단계** $\rightarrow$ 충돌 여부를 확인하는 **검증(Validation) 단계** $\rightarrow$ DB에 기록하는 **기록(Write) 단계**의 3단계 프로세스를 밟습니다.
     * *키워드*: 검증 없이 일단 수행, 판독-검증-기록 단계.
  2. **2단계 커밋 (2-Phase Commit / 2PC) (지문 ②번)**:
     * *개념*: **분산 데이터베이스 환경**에서 여러 노드에 걸쳐 있는 트랜잭션의 원자성(All-or-Nothing)과 일관성을 보장하기 위해 사용되는 동기화 프로토콜입니다.
     * *작동 단계*: 조정자(Coordinator)의 지휘 하에 전체 노드가 준비 상태를 응답하는 **1단계: 준비(Prepare) 단계** $\rightarrow$ 전원 반영 혹은 전체 취소를 결정하는 **2단계: 커밋(Commit) 단계**로 나누어 실행됩니다.
     * *주의*: 단일 DB에서 로킹 규약을 통해 직렬 가능성을 보장하는 **`2단계 로킹 (2-Phase Locking / 2PL)`**과 명확하게 구분해야 합니다! (분산 DB라는 단서가 있으면 무조건 `2단계 커밋`입니다.)

## 70. Python - 객체 지향 프로그래밍 (생성자 상속 및 클래스/인스턴스 변수 바인딩)

### 문제
아래 파이썬 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인의 빈칸 `def___________ __`과 오타는 정상 문법인 `def __init__(self, x):`으로 보정하여 풀이한다.)
```python
class A:
    x="ABC"
    def __init__(self, x):
        self.x=self.x+x
    def fn(self, x):
        print("{0}{1}".format(self.x, x))

class B(A):
    x="XYZ"
    def fn(self, x, y):
        print("{0}{1}{2}".format(self.x, x, y))

a=A("HI")
a.fn("J")

b=B("ST")
b.fn("UV", "W")
```

### 정답 및 풀이 결과
* **작성한 답**: `ABCHIJ\nXYZSTUVW` (자식 클래스 생성자 미정의 시 부모 생성자 상속 호출 메커니즘 및 클래스 변수 탐색 오버라이딩 완벽 추적 완료)
* **모범 답안**:
  ```text
  ABCHIJ
  XYZSTUVW
  ```
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
이 문제는 파이썬 클래스의 **생성자 상속 메커니즘**과 **클래스 변수 및 인스턴스 변수의 검색 바인딩 순서**를 묻는 수준 높은 문제입니다.

* **1단계: `a=A("HI")` 및 `a.fn("J")` (부모 클래스 인스턴스)**:
  * 객체 `a`가 생성되면서 `A` 클래스의 생성자 `__init__(self, x)`가 매개변수 `x="HI"`로 실행됩니다.
  * **`self.x = self.x + x`**: 
    * 우변의 `self.x` 평가 시, 아직 인스턴스 변수 `self.x`가 생성되지 않았으므로 **클래스 변수 `x`**인 `"ABC"`를 가져옵니다.
    * 이에 따라 `self.x = "ABC" + "HI"` = `"ABCHI"`가 되며, 이는 객체 `a`에 인스턴스 변수로 새로 저장됩니다.
  * **`a.fn("J")`**: 부모 클래스의 `fn`에 의해 인스턴스 변수 `"ABCHI"`와 인수 `"J"`가 더해져 **`"ABCHIJ"`**가 출력됩니다.
* **2단계: `b=B("ST")` 및 `b.fn("UV", "W")` (자식 클래스 인스턴스 - 핵심 함정)**:
  * **생성자 상속**: 자식 클래스 `B`에 생성자 `__init__`가 따로 선언되어 있지 않으므로, **부모 클래스 `A`의 `__init__` 생성자가 자동으로 호출**됩니다.
  * **`self.x = self.x + x` (인스턴스 `b`로 호출 시)**:
    * 우변의 `self.x` 평가 시, 인스턴스 변수가 없으므로 클래스 변수를 찾습니다.
    * 이때 현재 객체의 인스턴스 타입이 `B`이므로, 부모인 `A`가 아니라 **자식 클래스 `B`에 선언된 클래스 변수 `x`**인 `"XYZ"`를 가장 먼저 탐색하여 바인딩합니다!
    * 이에 따라 `self.x = "XYZ" + "ST"` = `"XYZST"`가 되어 자식 객체 `b` 내부에 인스턴스 변수로 새롭게 저장됩니다.
  * **`b.fn("UV", "W")`**: 자식 클래스 `B`에 오버라이딩된 `fn`에 의해 인스턴스 변수 `"XYZST"`와 매개변수 `"UV"`, `"W"`가 합쳐져 **`"XYZSTUVW"`**가 출력됩니다.

## 71. Java - 컬렉션 프레임워크 (HashSet 클래스 연산 및 toString() 출력 포맷)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 4라인 끝의 `；` 등 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```java
import java.util.Set;
import java.util.HashSet;

public class Soojebi {
    public static void main (String[] args){
        Set h=new HashSet();
        h.add(2);
        System.out.print(h);
        h.add(1);
        System.out.print(h);
        h.add(1);
        System.out.print(h);
        h.remove(1);
        System.out.print(h);
        System.out.print(h.size());
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `{2}{2,1}{2,1}{2}1` (HashSet의 중복 배제 및 remove 연산 흐름은 맞췄으나, 자바 컬렉션의 toString() 기본 출력 기호인 대괄호 `[]` 매핑과 원소 간 쉼표+공백 출력 포맷팅 오류로 감점 오답)
* **모범 답안**: **`[2][1, 2][1, 2][2]1`**
* **결과**: **오답 분석 및 Java 출력 포맷팅 규칙 보완 완료**

### 핵심 해설
이 문제는 **HashSet의 자료구조적 특징(중복 불허, 순서 없음)**과 **자바 컬렉션 객체를 직접 출력할 때의 문자열 변환(toString) 규칙**을 동시에 묻는 문제입니다.

* **1단계: HashSet의 연산 및 값 추적**:
  1. `Set h = new HashSet();` $\rightarrow$ 비어 있는 셋 객체 생성.
  2. `h.add(2);` $\rightarrow$ `h = [2]`
  3. `h.add(1);` $\rightarrow$ `h = [1, 2]` (해시코드 기준으로 정렬되어 내부적으로 `1`이 앞서 위치함)
  4. `h.add(1);` $\rightarrow$ 중복 값 추가이므로 셋의 특성상 무시됨. 여전히 `h = [1, 2]`.
  5. `h.remove(1);` $\rightarrow$ 원소 `1`을 제거하므로 `h = [2]`.
  6. `h.size();` $\rightarrow$ 남은 원소의 개수인 `1`을 반환.
* **2단계: 자바 컬렉션(List, Set)의 `toString()` 출력 포맷팅 규칙 (핵심 함정)**:
  * 파이썬은 집합(set) 출력 시 중괄호 `{ }`를 사용합니다.
  * 하지만 **자바의 컬렉션 클래스(HashSet, ArrayList 등)는 객체를 직접 출력할 때 대괄호 `[ ]`로 감싸서 문자열을 포맷팅**합니다.
  * 또한 원소와 원소 사이에는 **`쉼표와 한 칸 공백 (", ")`**이 자동으로 추가됩니다.
  * 이에 따라 각 단계별 출력 형식은 다음과 같습니다:
    * `System.out.print(h);` (2 추가 후) $\rightarrow$ **`[2]`**
    * `System.out.print(h);` (1 추가 후) $\rightarrow$ **`[1, 2]`**
    * `System.out.print(h);` (중복 1 추가 후) $\rightarrow$ **`[1, 2]`**
    * `System.out.print(h);` (1 삭제 후) $\rightarrow$ **`[2]`**
    * `System.out.print(h.size());` $\rightarrow$ **`1`**
* **최종 결합**:
  * 모든 출력문이 줄 바꿈이 없는 `System.out.print`이므로 옆으로 전부 붙여져서 최종적으로 **`[2][1, 2][1, 2][2]1`**이 출력됩니다.

* **💡 사용자 예리한 피드백 반영 (HashSet의 순서 무관성)**:
  * **컴퓨터 과학적 이론**: 집합(Set) 자료구조는 정의상 **순서가 없습니다 (Unordered)**. 따라서 이론적으로는 `[2, 1]`로 표기하든 `[1, 2]`로 표기하든 수학적으로 완벽히 동일한 집합입니다.
  * **자바 JVM 실측 런타임 결과**: 자바의 `Integer` 객체는 해시코드(`hashCode()`) 값으로 자기 자신의 정수 값을 그대로 반환합니다. 이로 인해 `HashSet` 내부 해시 테이블 버킷에 배치할 때 순차적인 버킷 인덱스를 얻게 되어, 실제 JVM 실행 시 toString() 결과물은 항상 **`[1, 2]`**로 일관되게 출력됩니다.
  * **수험 실전 채점 팁**: 정보처리기사 실기 시험은 기본적으로 **"자바 프로그램의 표준 출력(Standard Output) 결과"**를 100% 매칭 채점 기준으로 삼습니다. 따라서 채점관과의 정답 시비를 사전에 완벽히 방단(방지)하기 위해, JVM 런타임 기준인 **`[1, 2]`**로 답을 적어내는 것이 가장 안전하고 확실한 득점 전략입니다.

## 72. SQL - DML 작성 (GROUP BY 및 SUM 집계 함수를 이용한 그룹별 집계 쿼리)

### 문제
직원 테이블에서 부서별 급여의 합계를 구하는 쿼리를 작성하시오.
* **직원 테이블**: [이름, 부서, 급여]
* **결과 양식**: [부서, 급여합계]

### 정답 및 풀이 결과
* **작성한 답**: `select 부서, sum(급여) as 급여합계 from 직원 group by 부서;` (그룹 지정 컬럼, 집계 함수, 컬럼 별칭 및 GROUP BY 절 결합 무결하게 완수)
* **모범 답안**:
  ```sql
  SELECT 부서, SUM(급여) AS 급여합계 FROM 직원 GROUP BY 부서;
  ```
  *(※ 대소문자 무관 및 별칭 지정어 `AS` 생략 버전인 `SELECT 부서, SUM(급여) 급여합계 ...` 도 전원 정답 인정)*
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 SQL의 집계 연산 중 특정 기준 컬럼으로 행들을 묶어서 집계를 내는 **GROUP BY** 구문의 작성 능력을 평가하는 기본적이면서 핵심적인 기출 유형입니다.

* **쿼리 조밀 분석**:
  1. **`GROUP BY 부서`**: 부서별로 사원 데이터를 그룹화하여 묶어줍니다.
  2. **`SUM(급여) AS 급여합계`**: 묶인 각 부서 내 사원들의 급여 값을 합산한 뒤, 결과 릴레이션의 헤더 속성명을 맞추기 위해 별칭(Alias) 지시어 `AS`를 이용해 **`급여합계`**로 정의합니다.
  3. **`SELECT 부서, ...`**: 그룹화 기준이 된 속성(`부서`)과 집계된 속성을 SELECT 문에 나열합니다.
* **⚠️ GROUP BY 작성 시 절대 금기 (실제 시험 빈출 감점 요인)**:
  * **"SELECT 절에는 GROUP BY 절에 명시한 그룹화 기준 컬럼 또는 집계 함수(SUM, AVG 등)에 싸인 컬럼만 적을 수 있다!"**
  * 예를 들어, `SELECT 이름, 부서, SUM(급여) ... GROUP BY 부서;` 와 같이 작성하면 **문법 에러**가 납니다. '영업 부서' 그룹 내에 속한 사원의 이름이 '선조' 등 여러 명인데, 이를 단일 행으로 축약해서 표현할 방법이 없기 때문입니다. 이 문법적 한계를 명확히 기억해 두어야 합니다.

## 73. Java - 프로그래밍 언어 활용 (메서드 오버로딩 및 16진수 소문자 포맷 %x 출력)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인 끝의 `；` 등 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```java
public class Soojebi {
    static int func(int n){
        return n*9;
    }
    static int func(int a, int b){
        return a*b;
    }
    public static void main(String[] args){
        System.out.printf("%x", func(5)+func(2,3));
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `33` (인수 개수 기반의 오버로딩 매칭, 정수 대수 합산 및 %x 서식자의 접두어 배제 조건 완벽 인지 후 정답 도출 성공)
* **모범 답안**: **`33`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
이 문제는 **메서드 오버로딩(Overloading)**과 C/Java 계열의 **출력 포맷 지정자(Format Specifier)**의 세부 양식을 묻는 고전적인 복합 변별력 문제입니다.

* **1단계: 메서드 오버로딩 매칭 및 연산**:
  * **`func(5)`**: 인수가 1개이므로 `func(int n)` 메서드가 호출되어 `5 * 9 = 45`를 반환합니다.
  * **`func(2, 3)`**: 인수가 2개이므로 `func(int a, int b)` 메서드가 호출되어 `2 * 3 = 6`을 반환합니다.
  * **합산**: `45 + 6 = 51` (10진수)
* **2단계: 16진수 소문자 포맷(`%x`) 변환 (핵심 감점 함정)**:
  * 10진수 `51`을 16진수로 변환합니다:
    * $51 = 16 \times 3 + 3 \rightarrow$ 16진수 **`33`**
  * **서식 지정자 종류별 출력 규격**:
    * **`%x`**: 16진수 정수를 **소문자**로 출력합니다 (접두어 없음). $\rightarrow$ **`33`**
    * **`%X`**: 16진수 정수를 **대문자**로 출력합니다 (접두어 없음). $\rightarrow$ **`33`** (만약 알파벳 A~F가 섞여 있다면 대문자로 출력됨)
    * **`%#x`**: 16진수 소문자 표기 앞에 **접두어 `0x`**를 강제로 붙여서 출력합니다. $\rightarrow$ **`0x33`**
    * **`%#X`**: 16진수 대문자 표기 앞에 **접두어 `0X`**를 강제로 붙여서 출력합니다. $\rightarrow$ **`0X33`**
  * 본 지문은 단순 **`%x`**를 사용했으므로, 접두어 `0x` 없이 순수한 소문자 문자열인 **`33`**이 정답이 됩니다.

## 74. 데이터베이스 - 관계형 모델 설계 및 정규화(Normalization) 단계 판정 (제1정규형)

### 문제
다음과 같이 위쪽 릴레이션을 아래쪽 릴레이션으로 정규화를 하였을 때 어떤 정규화 작업을 한 것인지 쓰시오.

* **위쪽 릴레이션 (비원자값 존재)**:
  * 대한민국 $\rightarrow$ 서울, 부산
  * 미국 $\rightarrow$ 워싱턴, 뉴욕
* **아래쪽 릴레이션 (원자값 분해 완료)**:
  * 대한민국 $\rightarrow$ 서울 / 대한민국 $\rightarrow$ 부산
  * 미국 $\rightarrow$ 워싱턴 / 미국 $\rightarrow$ 뉴욕

### 정답 및 풀이 결과
* **작성한 답**: `제2 정규형` (테이블 모든 속성값이 쪼개지지 않는 최소 단위의 원자값을 가지는 제1정규형의 기본 정의 혼동으로 오답)
* **모범 답안**: **`제1정규형`** (또는 `1NF`, `1정규형`)
* **결과**: **오답 분석 및 정규화 6단계 암기 공식 보완 완료**

### 핵심 해설
이 문제는 관계형 데이터베이스 설계에서 이상 현상(Anomaly)을 방지하기 위한 **정규화(Normalization)의 각 단계별 이행 기준**을 묻는 매우 중요한 개념 문제입니다.

* **지문 릴레이션 변화 매커니즘 분석**:
  * **위쪽 릴레이션**: '대한민국' 부속 '도시' 속성에 '서울, 부산'이라는 **다중값(비원자값)**이 하나의 도메인 칸에 묶여 들어가 있습니다.
  * **아래쪽 릴레이션**: 다중값 속성을 개별 행으로 전부 쪼개어, 릴레이션 내 모든 속성의 값이 더 이상 분해될 수 없는 **원자값(Atomic Value)**만 가지도록 변환하였습니다.
  * **정의 연계**: 테이블의 모든 도메인이 원자값만으로 구성되도록 분해하는 정규화 과정은 **제1정규형 (1NF)**에 속합니다.
* **정규화 6단계 핵심 두문자 암기 공식 (기출 무조건 출제)**:
  * 🌟 **"도 $\rightarrow$ 부 $\rightarrow$ 이 $\rightarrow$ 결 $\rightarrow$ 다 $\rightarrow$ 조"**
    1. **`도`**메인이 원자값 $\rightarrow$ **제1정규형 (1NF)** (본 문제 해당)
    2. **`부`**분 함수적 종속성 제거 (완전 함수적 종속 만족) $\rightarrow$ **제2정규형 (2NF)**
    3. **`이`**행적 함수적 종속성 제거 ($A \rightarrow B, B \rightarrow C$ 일 때 $A \rightarrow C$ 제거) $\rightarrow$ **제3정규형 (3NF)**
    4. **`결`**정자이면서 후보키가 아닌 것 제거 (모든 결정자가 후보키) $\rightarrow$ **보이스-코드 정규형 (BCNF)**
    5. **`다`**치 종속 제거 $\rightarrow$ **제4정규형 (4NF)**
    6. **`조`**인 종속성 제거 (후보키를 통해서만 조인 종속 성립) $\rightarrow$ **제5정규형 (5NF)**

## 75. Java - 제어문 (for 반복문 탈출 조건 분석 및 경계값 연산)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인 끝의 `；` 및 4라인의 `〈=` 등 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```java
public class Soojebi {
    public static void main(String[] args){
        int i, j=0, sum=0;
        for(i=1; i<=110; i++){
            if(i%4==0){
                sum+=1;
                j=i;
            }
        }
        System.out.print(i +j+sum);
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `246` (for 루프 최종 탈출 시 인덱스 변수의 최종 1 증가 상태와 4의 배수 조건 완벽 추적 완료)
* **모범 답안**: **`246`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
이 문제는 **반복문이 정상적으로 끝난 뒤, 루프의 제어 인덱스 변수가 갖는 최종 경계값**을 정밀하게 디버깅할 수 있는지 검증하는 빈출 낚시 문제입니다.

* **각 변수의 동작 추적**:
  1. **`sum` (4의 배수 개수)**:
     * `i`가 `1`부터 `110`까지 1씩 증가하는 동안 `i % 4 == 0` (4의 배수) 조건이 만족되는 횟수입니다.
     * $110 \div 4 = 27$ (나머지 2) 이므로 총 개수 `sum = 27` 입니다.
  2. **`j` (가장 마지막에 매칭된 4의 배수)**:
     * 루프를 도는 동안 조건문이 참이 될 때마다 `j = i`로 갱신됩니다.
     * `1`부터 `110` 사이의 수 중 가장 마지막(가장 큰) 4의 배수는 $4 \times 27 = 108$ 입니다.
     * 따라서 루프를 완전히 다 돌았을 때 `j` 변수에는 **`108`**이 보존됩니다.
  3. **`i` (루프 제어 변수의 최종 탈출값 - 핵심)**:
     * 루프의 본문은 `i = 110`일 때까지 마지막으로 실행을 완료합니다.
     * 실행 직후 증감식인 **`i++`**에 의해 `i`는 **`111`**로 1 증가합니다.
     * 그 다음 조건식인 `i <= 110`을 판단하게 되는데, `111 <= 110`은 거짓(False)이 되므로 그제야 비로소 `for` 루프를 빠져나옵니다.
     * 따라서 루프가 종료된 시점에 변수 `i`의 최종값은 `110`이 아닌 **`111`**이 됩니다.
* **최종 덧셈 연산**:
  * `i + j + sum` = `111 + 108 + 27 = 246`이 출력됩니다.

## 76. 소프트웨어 프로젝트 관리 - 비용 산정 모형 연산 (Man-Month vs 개발 기간)

### 문제
LoC가 20,000라인이고, 개발자가 10명이며, 개발자는 월평균 100라인을 개발한다고 할 때 Man Month를 구하시오.

### 정답 및 풀이 결과
* **작성한 답**: `20` (총 노력인 Man-Month와 프로젝트 실제 개발 기간(Month)의 공식 요소를 혼동하여 아깝게 감점 오답)
* **모범 답안**: **`200`** (또는 `200 PM`, `200 인월`)
* **결과**: **오답 분석 및 비용 산정 공식 재정비 완료**

### 핵심 해설
이 문제는 소프트웨어 공학의 **상향식 비용 산정 기법 (LOC / Man-Month 기법)**에서 수험생들이 개념을 어설프게 알고 있을 때 저지르는 가장 흔한 낚시 문제입니다.

* **핵심 공식 구분**:
  1. **`Man-Month (총 노력 / Effort)`**:
     * 프로젝트를 완수하기 위해 필요한 **'총 작업량(인원수 × 개월수)'**을 뜻합니다.
     * 공식: **$\text{Man-Month} = \frac{\text{총 LOC}}{\text{개발자 1인당 월평균 개발 라인수(생산성)}}$**
     * 계산: $\frac{20,000}{100} = \mathbf{200}$
  2. **`개발 기간 (Duration / Month)`**:
     * 투입된 인원을 기준으로 프로젝트가 **'완료될 때까지 실제로 걸리는 개월 수'**를 뜻합니다.
     * 공식: **$\text{개발 기간 (Month)} = \frac{\text{Man-Month}}{\text{투입 개발자 수}}$**
     * 계산: $\frac{200}{10} = \mathbf{20}$
* **패인 진단**:
  * 문제에서 직접적으로 요구한 것은 **`Man Month`**의 수치이므로 정답은 **`200`**이 맞습니다.
  * 그러나 최종 계산 단계에서 개발자 수 `10`명으로 성급하게 한 번 더 나누어 **`개발 기간(20개월)`**을 계산하여 적어낸 것이 오답의 결정적 요인입니다. 문제의 발문이 **"Man-Month"**를 구하는 것인지 **"개발 기간"**을 구하는 것인지 마지막 단어를 극도로 경계하여 확인해야 합니다.

## 77. 소프트웨어 프로젝트 관리 - 일정 관리 모델 (CPM 및 PERT 기법)

### 문제
다음은 일정 관리에 대한 설명이다. 괄호 ( ) 안에 들어갈 올바른 용어를 쓰시오.
* 일정 관리 모델 종류 중에서 ( ① )은/는 여러 작업의 수행 순서가 얽혀 있는 프로젝트의 일정을 계산하는 기법으로 모든 자원 제약사항을 배제한 상태로 프로젝트의 시작과 끝을 나타내는 노드(Node)와 노드 간의 연결을 통해 공정을 계산하기 위한 액티비티(Activity) 표기법이고,
* ( ② )은/는 일의 순서를 계획적으로 정리하기 위한 수렴 기법으로 비관치, 중간치, 낙관치의 3점 추정 방식을 통해 일정을 관리하는 기법이다.

### 정답 및 풀이 결과
* **작성한 답**: 미작성 (프로젝트 관리 일정 수립 도구의 2대 대표 명칭 생소로 정밀 해설 요청)
* **모범 답안**:
  * **①: CPM (또는 임계 경로법, Critical Path Method)**
  * **②: PERT (또는 퍼트, Program Evaluation and Review Technique)**
* **결과**: **이론 완벽 정리 완료**

### 핵심 해설
이 문제는 소프트웨어 프로젝트 관리 중 **일정 계획 수립(Schedule Planning)** 단계에서 일정 네트워크를 구성하고 일정을 추정하는 대표적 양대 도구인 **CPM**과 **PERT**를 묻는 단골 기출 문제입니다.

* **지문 단서 매핑 및 개념 정립**:
  1. **CPM (Critical Path Method / 임계 경로법) (지문 ①번)**:
     * *개념*: 자원 제약은 배제하고, 프로젝트의 시작부터 끝까지 가장 긴 시간(여유 시간이 없는 경로)을 계산하여 프로젝트의 **최소 소요 기간(임계 경로)**을 찾아내는 일정 관리 도구입니다.
     * *키워드*: 자원 제약 배제, 노드와 액티비티 연결, 공정 계산.
  2. **PERT (Program Evaluation and Review Technique / 퍼트) (지문 ②번)**:
     * *개념*: 프로젝트의 각 작업 시간이 불확실할 때, **3점 추정치**를 이용하여 확률적으로 평균 소요 기간을 예측하는 기법입니다.
     * *🌟 PERT 3점 추정 평균 소요 시간 공식 (시험 단독 출제)*:
       $$\text{평균 소요 시간} = \frac{\text{낙관치} + (4 \times \text{최빈치(중간치)}) + \text{비관치}}{6}$$
     * *키워드*: 비관치-중간치-낙관치, 3점 추정 방식.

## 78. 네트워크 - 데이터 링크 계층(Data Link Layer)의 3대 핵심 제어 기능

### 문제
네트워크 계층에 데이터를 전달하고, 물리 계층에서 발생할 수 있는 오류를 탐지하고 수정하는 기능을 제공하는 데이터 링크 계층에 대한 설명이다. 괄호 ( ) 안에 올바른 용어를 쓰시오.
* ( ① )은/는 두 개의 스테이션이 동시에 신호를 전송하는 경우 신호 간 충돌이 발생하지 않도록 제어하는 기술로 ENQ/ACK 기법과 폴링 기법이 있다.
* ( ② )은/는 전송 스테이션으로 하여금 전송 데이터의 양을 제한하기 위해서 사용되는 기술로 정지-대기 기법과 슬라이딩 윈도우 기법이 있다.
* ( ③ )은/는 OSI 7 Layer의 하위의 두 계층 사이에서 데이터의 전송 오류를 검출하여 복구하는 기술로 해밍 코드와 같은 전진 오류 수정(FEC) 기법과 체크섬, CRC, ARQ와 같은 후진 오류 수정(BEC) 기법이 있다.

### 정답 및 풀이 결과
* **작성한 답**: 미작성 (데이터 링크 계층의 내부 제어 기술들의 명칭 매칭 생소로 상세 해설 분석 요청)
* **모범 답안**:
  * **①: 회선 제어 (Line Control)** (또는 `회선 제어 기법`)
  * **②: 흐름 제어 (Flow Control)** (또는 `흐름 제어 기법`)
  * **③: 오류 제어 (Error Control)** (또는 `오류 제어 기법`)
* **결과**: **이론 완벽 정리 완료**

### 핵심 해설
이 문제는 OSI 7계층 중 2계층인 **데이터 링크 계층 (Data Link Layer)**이 수행하는 가장 원초적이고 중요한 **3대 제어 기능**의 명칭과 그 하위 세부 기법들을 매칭시키는 네트워크 기초 개념 문제입니다.

* **3대 제어 기능 지문 핵심 단서 및 상세 매칭**:
  1. **회선 제어 (Line Control) (지문 ①번)**:
     * *개념*: 송신측과 수신측 간의 물리적 접속 전송 링크를 효율적으로 설정, 유지, 해제하기 위한 규약입니다.
     * *핵심 기법*: 
       * **`ENQ/ACK 기법`**: 1:1 점대점(Point-to-Point) 통신에서 송신측이 ENQ를 보내고 수신측이 ACK로 답하여 데이터를 주고받는 방식.
       * **`폴링(Polling) / 셀렉션(Selection) 기법`**: 1:N 다중점(Multipoint) 통신에서 주 컴퓨터가 제어권을 가지고 단말기들에게 송신 여부를 묻거나(Polling), 보낼 테니 받으라고 지목하는(Selection) 방식.
  2. **흐름 제어 (Flow Control) (지문 ②번)**:
     * *개념*: 송신측의 전송 속도가 너무 빨라 수신측의 버퍼 메모리가 넘쳐 데이터가 손실(Overflow)되는 현상을 막기 위해, **데이터 전송량을 조절**하는 제어 기술입니다.
     * *핵심 기법*:
       * **`정지-대기 (Stop-and-Wait)`**: 패킷 1개 보내고 수신측의 ACK를 받을 때까지 멈춰 서서 대기하는 방식.
       * **`슬라이딩 윈도우 (Sliding Window)`**: 수신측이 허용한 윈도우 크기(Window Size)만큼 ACK 없이 연속으로 패킷을 전송하는 효율적인 방식.
  3. **오류 제어 (Error Control) (지문 ③번)**:
     * *개념*: 데이터 전송 시 발생하는 비트 왜곡, 분실 등의 **오류를 감지하고 복구**하는 기술입니다.
     * *핵심 기법*:
       * **`전진 오류 수정 (FEC / Forward Error Correction)`**: 송신측이 오류 수정 코드(예: **해밍 코드**, 상승 코드)를 함께 보내 수신측이 스스로 오류를 찾아 고치는 방식.
       * **`후진 오류 수정 (BEC / Backward Error Correction)`**: 수신측이 오류를 발견하면 송신측에게 재전송을 요청하는 방식 (검출 기법: 패리티 비트, 체크섬, **CRC** / 재전송 요청 기법: **ARQ**).

## 79. C언어 - 수학적 알고리즘 (소수 판별 함수 및 범위 내 소수 개수 산출)

### 문제
아래 C 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인 끝의 `；` 등 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```c
#include <stdio.h>
int Soojebi(int num) {
    int i;
    for(i=2; i<num; i++){
        if(num % i==0)
            return 0;
    }
    return 1;
}
int main() {
    int num=7, cnt=0, i;
    for(i=2; i<num; i++)
        cnt+=Soojebi(i);
    printf("%d\n", cnt);
    return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `3` (소수 판별 메커니즘 해석 및 main 루프 범위 2 ~ 6 경계값 정확히 반영하여 정답 도출 성공)
* **모범 답안**: **`3`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
이 문제는 C언어를 활용한 **소수(Prime Number) 판별 알고리즘**의 해석 능력과 반복문의 수행 영역을 정밀하게 제어할 수 있는지 평가하는 문제입니다.

* **1단계: `Soojebi(num)` 함수 해석**:
  * 매개변수로 전달받은 정수 `num`에 대해, `2`부터 `num - 1`까지의 모든 수로 나누어 봅니다.
  * 만약 나누어떨어지는 제수 `i`가 발견되면 (`num % i == 0`) 즉시 `return 0;` (소수 아님)으로 종료됩니다.
  * 모든 나눗셈을 다 돌파하고 루프를 정상 탈출하면 `return 1;` (소수 맞음)을 반환합니다.
  * 즉, 이 함수는 전달된 인수가 소수이면 `1`, 소수가 아니면 `0`을 반환하는 함수입니다.
* **2단계: `main` 함수 연산 추적**:
  * `for(i=2; i<num; i++)`에 의해 `i`는 **`2, 3, 4, 5, 6`**까지 1씩 증가하며 반복됩니다. (상한선 `num = 7` 미만이므로 `7`은 루프 범위에서 제외됩니다.)
  * 각 단계별 `Soojebi(i)`의 반환값 및 `cnt` 누적 상태:
    * `i = 2`: 소수이므로 `1` 반환 $\rightarrow$ `cnt = 1`
    * `i = 3`: 소수이므로 `1` 반환 $\rightarrow$ `cnt = 2`
    * `i = 4`: 소수 아님(2로 나누어떨어짐) $\rightarrow$ `0` 반환 $\rightarrow$ `cnt = 2`
    * `i = 5`: 소수이므로 `1` 반환 $\rightarrow$ `cnt = 3`
    * `i = 6`: 소수 아님(2, 3으로 나누어떨어짐) $\rightarrow$ `0` 반환 $\rightarrow$ `cnt = 3`
* **최종 결과**:
  * `i = 7`이 되면 조건식 `i < num` (`7 < 7`)이 거짓이 되므로 루프를 종료하고 최종 `cnt` 값인 **`3`**을 줄 바꿈과 함께 출력합니다.

## 80. 데이터베이스 - 트랜잭션 제어(Transaction Control) 연산 (COMMIT 및 ROLLBACK)

### 문제
다음은 트랜잭션 연산에 대한 설명이다. 괄호 ( ) 안에 들어갈 명령어를 쓰시오.
* ( ① )은/는 하나의 트랜잭션이 성공적으로 끝나고, 데이터베이스가 일관성 있는 상태에 있거나 하나의 트랜잭션이 끝났을 때 사용하는 연산이다.
* ( ② )은/는 하나의 트랜잭션이 비정상적으로 종료되어 트랜잭션 원자성이 깨질 경우 모든 변경사항이 취소되고, 트랜잭션이 시작되기 전의 상태로 되돌리는 연산이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 커밋 (COMMIT)**
  * **②: 롤백 (ROLLBACK)**
* **모범 답안**:
  * **①: COMMIT (또는 커밋)**
  * **②: ROLLBACK (또는 롤백)**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 데이터베이스의 논리적 작업 단위인 **트랜잭션(Transaction)의 상태 변화와 무결성 제어**를 담당하는 TCL(Transaction Control Language)의 양대 기본 연산어를 묻는 기출 빈출 문제입니다.

* **지문 단서와 명확한 매핑**:
  1. **COMMIT (커밋) (지문 ①번)**:
     * *개념*: 트랜잭션 내의 모든 변경 작업이 성공적으로 완수되어, 이 데이터 상태를 데이터베이스에 영구적으로 반영(기록)하는 확정 선언 연산입니다.
     * *의의*: COMMIT이 수행되고 나면, 변경 사항은 영구 보존되며 다른 트랜잭션들도 이 결과를 참조할 수 있게 됩니다.
  2. **ROLLBACK (롤백) (지문 ②번)**:
     * *개념*: 트랜잭션 실행 중 비정상적인 버그나 강제 중단이 일어났을 때, 변경 사항을 완전히 무효화하고 **트랜잭션 실행 직전 상태(세이브 포인트 또는 트랜잭션 시작점)로 되돌리는(Undo)** 취소 연산입니다.
     * *의의*: 이 연산을 통해 트랜잭션의 핵심 4대 속성인 **원자성(Atomicity / All-or-Nothing)**이 완벽하게 보장될 수 있습니다.

## 81. 소프트웨어 개발 보안 - 입력 데이터 검증 및 표현 취약점 (XSS 및 SQL Injection)

### 문제
다음은 입력 데이터 검증 및 표현 취약점에 대한 설명이다. 괄호 ( ) 안에 들어갈 공격 기법을 쓰시오.
* ( ① )은/는 검증되지 않은 외부 입력 데이터가 포함된 웹페이지가 전송되는 경우, 사용자가 해당 웹페이지를 열람함으로써 웹페이지에 포함된 부적절한 스크립트가 실행되는 공격 기법이다.
* ( ② )은/는 응용 프로그램의 보안 취약점을 이용해서 악의적인 SQL 구문을 삽입, 실행시켜서 데이터베이스(DB)의 접근을 통해 정보를 탈취하거나 조작 등의 행위를 하는 공격 기법이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 리디렉션** (웹페이지 강제 이동 기법과 악성 스크립트 강제 브라우저 실행 기법의 용어 혼동으로 오답)
  * **②: sql 삽입** (SQL Injection을 완벽하게 맞춤)
* **모범 답안**:
  * **①: 크로스 사이트 스크립팅 (또는 XSS, Cross-Site Scripting)**
  * **②: SQL 삽입 (또는 SQL Injection, SQL 인젝션)**
* **결과**: **부분 수정 후 최종 정답 도출 가능**

### 핵심 해설
이 문제는 정보보안 및 개발 보안 단원에서 **가장 출제율이 높고 기본이 되는 2대 웹 취약점**을 구별하는 문제입니다.

* **세부 취약점별 메커니즘 분석**:
  1. **크로스 사이트 스크립팅 (Cross-Site Scripting / XSS) (지문 ①번)**:
     * *개념*: 공격자가 웹사이트에 악성 스크립트(보통 자바스크립트)를 삽입해 두고, **일반 사용자가 이 웹페이지를 열람할 때 그 사용자의 웹 브라우저에서 해당 악성 스크립트가 자동 실행**되도록 유도하는 기법입니다.
     * *목적*: 세션 쿠키 탈취, 시스템 악성코드 유포, 피싱 사이트 우회 등.
     * *보정 피드백*: '리디렉션(Redirection)'은 사용자를 다른 URL로 돌리는 행위 자체를 지칭할 뿐이고, 취약점 지문에서 설명하는 **"부적절한 스크립트가 웹페이지 내에서 강제 실행"**되는 보안 공격 기법은 무조건 **`XSS`**입니다.
  2. **SQL 삽입 (SQL Injection) (지문 ②번)**:
     * *개념*: 웹 애플리케이션의 입력창(예: 로그인 창, 게시판 검색창 등)에 **악의적인 SQL 쿼리 조건문**(`' OR '1'='1`)을 주입하여, 백엔드 데이터베이스가 원래 설계와 다르게 승인되지 않은 정보를 다 토해내게 만드는 공격입니다.
     * *방어 대책*: 입력값 검증, 바인딩 변수 사용, 동적 쿼리 배제.
