## 50. 네트워크 - 라우팅 프로토콜 알고리즘 (거리 벡터 vs 링크 상태)

### 문제
다음이 설명하는 라우팅 프로토콜 알고리즘은 무엇인지 쓰시오.
* 인접 라우터와 정보를 공유하여 목적지까지의 거리와 방향을 결정하는 라우팅 프로토콜 알고리즘
* 벨만-포드(Bellman-Ford) 알고리즘 사용
* 각 라우터가 업데이트될 경우마다 전체 라우팅 테이블을 보내라고 요청하지만 수신된 경로 비용 정보는 이웃 라우터에게만 보냄

### 정답 및 풀이 결과
* **작성한 답**: `거리 벡터 알고리즘` (세부 매커니즘 및 링크 상태 알고리즘과의 비교 분석 질의 포함)
* **모범 답안**: **`거리 벡터 알고리즘 (Distance Vector Algorithm)`** (또는 `거리 벡터`)
* **결과**: **정답 (100% 매칭 및 상세 비교 분석 완료)**

### 핵심 해설
* **거리 벡터 (Distance Vector) 알고리즘**:
  * **핵심 철학 (소문 전파 방식)**: 목적지까지 도달하는 데 거치는 관문(Hop) 수인 **거리(Distance)**와 어느 다음 라우터로 가야 하는지인 **방향(Vector)** 정보만을 관리하는 직관적인 라우팅 방식입니다.
  * **수학적 기반**: 최단 경로 탐색을 위해 **벨만-포드 (Bellman-Ford) 알고리즘**을 사용합니다.
  * **전파 방식**: 전체 라우팅 정보를 **인접한 이웃 라우터(Direct Neighbors)**들과만 공유하며, 변경 여부와 관계없이 주기적으로 전체 라우팅 테이블을 갱신합니다.
* **시험에 무조건 대조되어 나오는 '링크 상태(Link State) 알고리즘' 비교**:
  1. **거리 벡터 알고리즘**:
     * **알고리즘**: 벨만-포드 (`Bellman-Ford`)
     * **전파 대상**: 인접 라우터만 (소문에 의해 갱신)
     * **전송 정보**: 전체 라우팅 테이블 통째로 전송
     * **대표 프로토콜**: **`RIP`**, **`IGRP`**
  2. **링크 상태 알고리즘**:
     * **알고리즘**: 다익스트라 (`Dijkstra` - 최단 경로 우선)
     * **전파 대상**: 네트워크 내의 모든 라우터 (Flooding)
     * **전송 정보**: 링크 상태 변경이 있을 때만 최소한의 변경 정보(LSA) 전송
     * **대표 프로토콜**: **`OSPF`**, **`IS-IS`**

## 51. Python - 문자열 슬라이싱(Slicing) 및 서식(Formatting) 연산

### 문제
아래 파이썬 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 1라인 끝의 `；` 등 OCR 오타는 무시하고 올바른 파이썬 문법으로 간주하여 풀이한다.)
```python
a="%s%s"% ("soojebi"[0:2], "soojebi"[3:4])
b=1111
print("%s%d"%(a*2, b*2))
```

### 정답 및 풀이 결과
* **작성한 답**: `sojsoj2222` (파이썬 슬라이싱 인덱스 범위, 자료형별 곱셈 규칙 및 서식 지정자 매핑 완벽 적용)
* **모범 답안**: **`sojsoj2222`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **문자열 슬라이싱(`[start:end]`) 분석**:
  * 파이썬의 슬라이싱은 `start` 인덱스부터 `end - 1` 인덱스까지의 문자열을 추출합니다.
  * `"soojebi"[0:2]`: `0`번 문자 `'s'`부터 `1`번 문자 `'o'`까지 추출되어 **`"so"`**가 됩니다. (2번 문자는 제외)
  * `"soojebi"[3:4]`: `3`번 문자인 **`"j"`**만 추출됩니다. (4번 문자는 제외)
    * *인덱스 매핑*: `s(0), o(1), o(2), j(3), e(4), b(5), i(6)`
  * 따라서, `a = "%s%s" % ("so", "j")`에 의해 변수 `a`에는 문자열 **`"soj"`**가 대입됩니다.
* **자료형에 따른 곱셈(`*`) 연산 차이**:
  * **문자열 곱셈(`a * 2`)**: 문자열 `"soj"`를 2회 반복 접합하므로 **`"sojsoj"`**가 됩니다.
  * **정수형 곱셈(`b * 2`)**: 정수 `1111`에 2를 곱하는 대수학 연산이므로 결과는 **`2222`**가 됩니다.
* **포맷 스트링 출력**:
  * `"%s%d" % ("sojsoj", 2222)`에 의해 앞은 문자열 형식(`%s`), 뒤는 10진 정수 형식(`%d`)으로 붙어서 출력되므로 최종 출력값은 **`sojsoj2222`**가 됩니다.

## 52. Python - 등비수열(Geometric Progression) 연산 루프 제어

### 문제
아래 파이썬 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 4라인의 `True：` 등 문장 부호 OCR 오타와 7라인의 `ifcnt= = n：` 은 `if cnt == n:` 으로 보정하고 들여쓰기는 올바르다고 가정하여 풀이한다.)
```python
def soojebi(begin, diff, n):
    cnt=1
    ret= begin
    while True:
        cnt+=1
        ret*=diff
        if cnt == n:
            return ret

print(soojebi(1, 3, 4))
```

### 정답 및 풀이 결과
* **작성한 답**: `27` (등비수열 루프 횟수 및 갱신 조건을 정확히 연산하여 풀이 성공)
* **모범 답안**: **`27`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **함수 매개변수 바인딩**:
  * `soojebi(1, 3, 4)`에 의해 `begin = 1` (초기 항), `diff = 3` (공비), `n = 4` (구하고자 하는 항 번호)가 대입됩니다.
* **디버깅 실행 순서 상세 추적**:
  * **초기화**: `cnt = 1`, `ret = 1`
  * **1회전**:
    * `cnt`가 1 증가하여 `2`가 됩니다.
    * `ret`에 공비 `3`을 곱하여 `1 * 3 = 3`이 됩니다.
    * 탈출 조건 검사: `cnt == n` $\rightarrow$ `2 == 4` (거짓, 계속 진행)
  * **2회전**:
    * `cnt`가 1 증가하여 `3`가 됩니다.
    * `ret`에 공비 `3`을 곱하여 `3 * 3 = 9`가 됩니다.
    * 탈출 조건 검사: `cnt == n` $\rightarrow$ `3 == 4` (거짓, 계속 진행)
  * **3회전**:
    * `cnt`가 1 증가하여 `4`가 됩니다.
    * `ret`에 공비 `3`을 곱하여 `9 * 3 = 27`이 됩니다.
    * 탈출 조건 검사: `cnt == n` $\rightarrow$ `4 == 4` (참, 리턴 및 탈출)
* **최종 출력**:
  * 함수가 최종 반환한 값인 **`27`**을 `print`문으로 화면에 출력하므로 결과는 `27`입니다.

## 53. C언어 - 구조체(Structure) 배열 및 실수 소수점 포맷 출력

### 문제
아래 C 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 1라인의 `ffinclude`는 `#include`로, 16라인의 `printf(“%d：%.2f s[i].no, s[i].d)；` 는 큰따옴표 누락을 보정하여 `printf("%d:%.2f ", s[i].no, s[i].d);` 로 간주하고 풀이한다.)
```c
#include <stdio.h>
struct soojebi {
    int no;
    int a, b, c;
    double d; 
};
struct soojebi s[3]={
    {1, 80, 90, 100},
    {2, 85, 95, 90},
    {3, 70, 80, 75}
};
int main() {
    int i;
    for(i=0; i<3; i++){
        s[i].d=(s[i].a+s[i].b+s[i].c)/3.0;
        printf("%d:%.2f ", s[i].no, s[i].d);
    }
    return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `1:90.00 2:90.00 3:75.00` (구조체 멤버별 평균 연산, 소수점 둘째 자리 포맷팅 및 한 줄 연속 출력 규칙 정확히 해석)
* **모범 답안**: **`1:90.00 2:90.00 3:75.00 `** (마지막 공백 포함 여부와 관계없이 정답 처리)
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **구조체 정의 및 배열 초기화**:
  * 구조체 `soojebi`는 5개의 멤버 변수(`no`, `a`, `b`, `c`, `d`)로 구성되어 있습니다.
  * 크기가 3인 구조체 배열 `s`를 선언할 때, `{no, a, b, c}` 값만 기입하여 `d` 값은 기본값인 `0.0`으로 자동 초기화됩니다.
* **평균 연산 (`s[i].d`) 상세 디버깅**:
  * **`i = 0`**:
    * `s[0].d = (80 + 90 + 100) / 3.0 = 270 / 3.0 = 90.0`
    * 출력: `printf("%d:%.2f ", 1, 90.0) -> "1:90.00 "`
  * **`i = 1`**:
    * `s[1].d = (85 + 95 + 90) / 3.0 = 270 / 3.0 = 90.0`
    * 출력: `printf("%d:%.2f ", 2, 90.0) -> "2:90.00 "`
  * **`i = 2`**:
    * `s[2].d = (70 + 80 + 75) / 3.0 = 225 / 3.0 = 75.0`
    * 출력: `printf("%d:%.2f ", 3, 75.0) -> "3:75.00 "`
* **서식 문자열 `%.2f`와 출력 서식 분석**:
  * `%.2f`는 실수를 소수점 아래 둘째 자리까지 반올림하여 표시합니다.
  * 포맷 스트링 내의 콜론(`:`)과 뒤쪽의 한 칸 공백(` `)으로 인해 개행 없이 한 줄에 공백을 두고 나란히 출력됩니다.

## 54. 데이터베이스 - 이벤트 기반 절차형 SQL 객체 (트리거 / Trigger)

### 문제
데이터베이스 시스템에서 삽입, 갱신, 삭제 등의 이벤트가 발생할 때마다 관련 작업이 자동으로 수행되는 절차형 SQL은 무엇인지 쓰시오.

### 정답 및 풀이 결과
* **작성한 답**: `트리거` (이벤트 기반 자동 실행 특징을 정확히 식별 성공)
* **모범 답안**: **`트리거 (Trigger)`**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
* **트리거(Trigger)의 설계 목적**:
  * 데이터베이스의 테이블에 `INSERT, UPDATE, DELETE` 등의 DML 문이 실행될 때, 해당 데이터의 변경을 방지하거나 무결성 유지를 위해 **연쇄 조작이 자동으로 작동**하도록 설정된 데이터베이스 객체입니다.
* **시험에 출제되는 '절차형 SQL 3대장' 핵심 특징 비교**:
  1. **프로시저 (Procedure)**:
     * 호출: `CALL 프로시저명;` 을 통한 **수동(명시적) 호출**
     * 반환값: 여러 개의 값 반환 가능 (IN/OUT 파라미터 사용)
     * 트랜잭션: 내부에서 `COMMIT` 및 `ROLLBACK` **제어 가능**
  2. **사용자 정의 함수 (User Defined Function)**:
     * 호출: `SELECT 함수명(...) FROM ...;` 쿼리문 내부에서 **식의 일부로 호출**
     * 반환값: **단 하나의 값**만 필수 반환 (RETURN 문 필수)
     * 트랜잭션: 내부에서 `COMMIT` 및 `ROLLBACK` **제어 불가**
  3. **트리거 (Trigger)**:
     * 호출: 이벤트 발생 시 시스템에 의해 **자동(암묵적)으로 가동**
     * 반환값: **반환값이 없음** (RETURN 문이 존재하지 않음)
     * 트랜잭션: 내부에서 `COMMIT` 및 `ROLLBACK` **제어 불가** (부모 트랜잭션과 한 생명주기로 작동하기 때문)

## 55. Java - 클래스 상속 및 다형성과 생성자 제약 조건 (컴파일 에러 분석)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 17라인의 `Parent c三new Child` 등 OCR 오타는 `Parent c = new Child("Soojebi");` 와 같이 정상적인 문법으로 보정하여 풀이한다.)
```java
class Parent{
    String name="Parent";
    public Parent(String name){
        System.out.print("A");
    }
}
class Child extends Parent{
    public Child(String name){
        System.out.print("B");
    }
    void info(){
        System.out.print(name); 
    }
}
public class Soojebi{
    public static void main(String[] args) {
        Parent c = new Child("Soojebi");
        c.info();
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `컴파일 에러` (오리지널 출제 답안 규격인 '부모 클래스 기본 생성자 부재로 인한 상위 클래스 생성자 호출 오류' 매커니즘 정밀 검토 완료)
* **모범 답안**:
  * **`Parent 클래스에서 매개변수가 없는 생성자가 존재하지 않기 때문에 에러가 발생한다.`**
  * (또는 **`Child 클래스에서 상위 클래스 Parent의 기본 생성자가 호출되지 않아 오류가 발생한다.`**)
* **결과**: **정답 (출제자의 의도 및 컴파일러 파싱 우선순위 완벽 매칭)**

### 핵심 해설
자바 컴파일러는 위에서 아래로 소스 코드를 읽어내려가며 검증합니다. 따라서 본 코드에서 **가장 먼저(1차로) 발생하는 에러의 근원(Root Cause)은 상속 구조에서의 생성자 호출 규칙 위반**입니다.

1. **상속 시 자식 생성자의 부모 생성자 호출 의무 (1차 오류 지점)**:
   * 자바에서 자식 클래스(`Child`)의 생성자 첫 줄에는 부모의 기본 생성자를 호출하는 `super();`가 컴파일러에 의해 자동으로 주입됩니다.
   * 그러나 부모 클래스 `Parent`에는 매개변수가 있는 생성자 `public Parent(String name)`만 선언되어 있어, 매개변수가 없는 디폴트 생성자 `Parent()`가 존재하지 않습니다.
   * 이에 따라 `Child` 클래스의 정의 부분(`7~14라인`)을 컴파일하는 단계에서 **"부모의 기본 생성자를 찾을 수 없다"**는 오류가 발생하여 즉시 컴파일이 중단됩니다.
   * **이것이 출제자가 의도한 최우선의 컴파일 오류 원인입니다.**
2. **다형성 참조 변수의 자식 고유 메서드 호출 오류 (2차 오류 지점)**:
   * 만약 부모 클래스에 기본 생성자를 보완하여 1차 에러를 넘기더라도, `main` 메서드 내부의 `Parent c = new Child("Soojebi");`에서 변수 `c`가 부모(`Parent`) 타입으로 선언되어 있기 때문에, 부모 클래스에 없는 자식 클래스의 고유 메서드인 `info()`를 호출할 때 2차 컴파일 에러가 연이어 터지게 됩니다.

* **💡 실전 서술형 답안 작성 팁**:
  * 문제집이나 실제 시험장에서 오류 원인을 서술할 때, **컴파일 단계상 1차적으로 컴파일러가 중단되는 '부모의 기본 생성자 부재'를 원인으로 적어주시는 것이 채점관의 기준표와 100% 합치**됩니다.
  * **공식 모범 답안**: "Parent 클래스에서 매개변수가 없는 생성자가 존재하지 않기 때문입니다." (또는 "Child 클래스에서 상위 클래스 Parent의 기본 생성자가 호출되지 않기 때문입니다.")

## 56. SQL DDL - 가상 테이블 뷰 생성 (CREATE VIEW AS)

### 문제
다음 사원 테이블 스키마를 보고, 사원 테이블에서 성별 값이 ‘M’을 가진 사원의 사번, 이름을 출력하는 ‘사원뷰’라는 이름의 뷰를 생성하는 쿼리를 작성하시오.
* 사원 테이블(사번, 업무, 이름, 생년월일, 성별, 입사일)

### 정답 및 풀이 결과
* **작성한 답**: `create view 사원 as select 사번, 이름 from 사원 where 성별 = 'M';` (SELECT 및 AS 뷰 생성 구조는 맞추었으나, 지정된 뷰 이름인 '사원뷰' 대신 원본 테이블명인 '사원'을 기재하는 미세 매핑 실수 발생)
* **모범 답안**:
  * **`CREATE VIEW 사원뷰 AS SELECT 사번, 이름 FROM 사원 WHERE 성별 = 'M';`** (또는 세미콜론 생략 및 소문자 기입도 정답)
* **결과**: **부분 수정 후 완벽 정답 도출 가능**

### 핵심 해설
* **뷰(View)의 개념**:
  * 물리적인 데이터를 보관하지 않고, 하나 이상의 테이블에서 유도된 가상 테이블(Virtual Table)입니다.
* **뷰 생성 DDL 문법 구조**:
  * **`CREATE VIEW 뷰_이름 AS SELECT ...`**
  * 이 구조에서 `AS` 키워드를 빼먹어 오답 처리되는 경우가 매우 흔하므로 주의해야 합니다.
* **지문 조건 정밀 매핑 (실수 요인 분석)**:
  * **뷰 이름 지정**: 지문에서 분명 **"‘사원뷰’라는 이름의 뷰"**를 만들라고 지시했습니다. 
  * 만약 작성자님 답안처럼 `CREATE VIEW 사원`으로 작성하게 되면, 이미 물리적으로 존재하는 원본 테이블 `사원`과 이름이 겹쳐서 데이터베이스에서 명칭 충돌 오류가 발생합니다.
  * 따라서 뷰의 이름인 **`사원뷰`**를 반드시 DDL 생성 명칭에 넣어주어야 합니다.

## 57. 소프트웨어 설계 - 객체지향 설계 5대 원칙 (SOLID)

### 문제
객체지향 설계 원칙은 5가지가 있다. 다음 지문의 괄호 ①, ② 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① )은 객체가 다른 클래스를 참조할 때 해당 클래스를 직접 참조하지 말고, 상위 요소인 추상 클래스나 인터페이스를 통해 참조하라는 원칙이다.
* ( ② )은 하위 클래스(서브 타입)는 언제든지 상위 클래스(기반 타입)로 대체할 수 있어야 한다는 원칙이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 의존성 역전의 원칙** (또는 `의존성 역전 원칙`, `DIP`)
  * **②: 리스코프 치환의 원칙** (또는 `리스코프 치환 원칙`, `LSP`)
* **모범 답안**:
  * **①: 의존성 역전 원칙 (Dependency Inversion Principle)**
  * **②: 리스코프 치환 원칙 (Liskov Substitution Principle)**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
* **객체지향 설계 5대 원칙 (SOLID)**:
  1. **S - 단일 책임 원칙 (Single Responsibility Principle)**:
     * *정의*: 하나의 클래스는 하나의 책임을 가지며, 클래스를 변경하는 이유는 단 하나여야 합니다.
  2. **O - 개방-폐쇄 원칙 (Open-Closed Principle)**:
     * *정의*: 확장에는 열려 있고(Open), 변경에는 닫혀(Closed) 있어야 합니다.
  3. **L - 리스코프 치환 원칙 (Liskov Substitution Principle) (지문 ②번)**:
     * *정의*: 서브 타입은 언제나 기반 타입으로 교체할 수 있어야 합니다. (부모를 자식으로 교체해도 프로그램이 오작동하지 않는 상속 계약 관계 성립)
  4. **I - 인터페이스 분리 원칙 (Interface Segregation Principle)**:
     * *정의*: 자신이 사용하지 않는 메서드에 의존 관계를 맺지 않아야 합니다. (범용 인터페이스 하나보다는 작고 구체적인 인터페이스 여러 개가 좋습니다.)
  5. **D - 의존성 역전 원칙 (Dependency Inversion Principle) (지문 ①번)**:
     * *정의*: 고수준 모듈은 저수준 모듈의 구체 클래스에 의존하지 않고, 인터페이스나 추상 클래스(추상화)에 의존해야 합니다.

## 58. 소프트웨어 개발 보안 - 보안 위협 및 취약점 공격 기법 (제로데이 공격)

### 문제
다음은 보안 관련 용어에 대한 설명이다. 괄호 ( ) 안에 공통으로 들어갈 용어는 무엇인지 쓰시오.
* ( )은/는 보안 취약점이 발견되어 널리 공표되기 전에 해당 취약점을 악용하여 이루어지는 보안 공격이다. 공격의 신속성을 의미하는 것으로, 일반적으로 컴퓨터에서 취약점이 발견되면 제작자나 개발자가 취약점을 보완하는 패치를 배포하고 사용자가 이를 다운받아 대처하지만, ( )은/는 대응책이 공표되기도 전에 공격이 이루어지기 때문에 대처 방법이 없다.

### 정답 및 풀이 결과
* **작성한 답**: `제로데이 공격` (취약점 패치 공표 전 감행하는 특성 정확히 캐치 성공)
* **모범 답안**: **`제로데이 공격 (Zero-Day Attack)`**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
* **제로데이 공격 (Zero-Day Attack)**:
  * **핵심 원리**: 보안 취약점의 발견일부터 시작하여 대처 방안(보안 패치, 업데이트)이 릴리즈되는 날 사이의 **공백 기간(Zero Day)**을 틈타 기습적으로 치명적인 타격을 입히는 공격 기법입니다.
  * **지문 단서**: "취약점이 발견되어 공표되기 전", "패치 배포 전 대처 방법 없음" $\rightarrow$ **`제로데이`**의 고유 지문입니다.
* **함께 묶여서 출제되는 '최신 핵심 보안 위협 용어 5선' (중요)**:
  1. **워터링 홀 (Watering Hole)**:
     * *개념*: 표적 집단이 주로 방문하는 합법적인 웹사이트를 감염시킨 뒤 잠복하여, 표적이 방문했을 때 자동으로 악성코드를 다운로드하게 유도하는 표적 공격 기법입니다.
  2. **APT (지능형 지속 위협 / Advanced Persistent Threat)**:
     * *개념*: 특정 대상을 목표로 삼고 장기간에 걸쳐 다양한 해킹 기법을 은밀하고 지속적으로 동원하여 침투하는 수법입니다.
  3. **드라이브 바이 다운로드 (Drive-by Download)**:
     * *개념*: 사용자가 눈치채지 못하게 악성코드가 유포되는 사이트를 방문하는 행위만으로 기기에 악성코드가 자동으로 다운로드되고 실행되게 만드는 공격입니다.
  4. **스피어 피싱 (Spear Phishing)**:
     * *개념*: 신뢰할 수 있는 송신자로 가장하여 특정 개인 또는 조직을 표적으로 유도성 메일을 발송해 금융/인증 정보를 훔치는 작살형 피싱 수법입니다.
  5. **랜섬웨어 (Ransomware)**:
     * *개념*: 사용자 시스템의 데이터를 암호화하여 몸값(Ransom)을 요구하는 대표적인 금전 목적의 악성코드입니다.

## 59. 소프트웨어 프로젝트 관리 - 상향식 비용 산정 모형 (Man-Month / COCOMO)

### 문제
다음은 비용 산정 모형에 대한 설명이다. 괄호 ( ) 안에 알맞은 모형은 무엇인지 쓰시오.
* ( ① )모형은 한 사람이 1개월 동안 할 수 있는 일의 양을 기준으로 프로젝트 비용을 산정하는 방식으로 원시 코드 라인 수인 LoC를 프로그래머의 월간 생산성으로 나누어 계산한다.
* ( ② )모형은 보헴(Boehm)이 제안한 프로그램 규모에 따라 비용을 산정하는 방식으로, 규모에 따라 조직형, 반 분리형, 임베디드형으로 나뉘는 모형이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 맨먼스** (또는 `Man-Month`, `M/M`, `인월`)
  * **②: COCOMO** (또는 `코코모`)
* **모범 답안**:
  * **①: Man-Month** (또는 `맨먼스`, `인월`)
  * **②: COCOMO** (또는 `코코모`)
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
* **Man-Month (인월 / M/M) 모형 (지문 ①번)**:
  * **개념**: 프로젝트에 필요한 총 노력을 1명의 개발자가 1개월 동안 수행할 수 있는 작업량으로 수치화한 것입니다.
  * **수식**:
    * $\text{Effort (Man-Month)} = \frac{\text{총 LOC}}{\text{프로그래머의 월간 생산성}}$
    * $\text{개발 기간 (Month)} = \frac{\text{Man-Month}}{\text{참여 프로그래머 수}}$
* **COCOMO (Constructive Cost Model) 모형 (지문 ②번)**:
  * **개념**: 보헴(Boehm)이 주창한 원시 코드 라인 수(LOC) 기반의 수학적 비용 산정 모형입니다.
  * **🌟 기출 핵심: COCOMO의 개발 규모별 3가지 구축 유형**:
    1. **조직형 (Organic Mode)**:
       * **규모**: **5만 라인(50KLOC) 이하**의 중소규모 사무용 시스템 개발에 적합합니다.
    2. **반 분리형 (Semi-Detached Mode)**:
       * **규모**: **30만 라인(300KLOC) 이하**의 컴파일러, 데이터베이스 제어 시스템 등 중규모 개발에 적합합니다.
    3. **임베디드형 (Embedded Mode)**:
       * **규모**: **30만 라인(300KLOC) 이상**의 초대형 시스템 또는 실시간 정밀 하드웨어 제어(항공, 국방 등) 시스템 개발에 적합합니다.

## 60. Java 제어문 - while 반복문과 continue 조건 분기 제어 흐름

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인의 `int i=0, sum=0；` 및 8라인의 `sum+三 i；` 와 같은 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```java
public class Soojebi {
    public static void main(String[] args){
        int i=0, sum=0;
        while(i<10){
            i++;
            if(i%2==1)
                continue;
            sum += i;
        }
        System.out.print(sum);
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `30` (10 이하의 짝수 합산 알고리즘 및 while 루프 탈출 조건 정밀 분석 성공)
* **모범 답안**: **`30`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **`while(i < 10)` 루프 및 변수 변화 추적**:
  * 초기 조건: `i = 0`, `sum = 0`
  * **루프 1회전 (i=0)**: `i++`에 의해 `i = 1`. `i % 2 == 1` 홀수이므로 `continue` 가동 $\rightarrow$ 루프 탑으로 회귀. (`sum = 0`)
  * **루프 2회전 (i=1)**: `i++`에 의해 `i = 2`. 짝수이므로 `sum += 2` 누적. (`sum = 2`)
  * **루프 3회전 (i=2)**: `i++`에 의해 `i = 3`. 홀수이므로 `continue` 가동.
  * **루프 4회전 (i=3)**: `i++`에 의해 `i = 4`. 짝수이므로 `sum += 4` 누적. (`sum = 6`)
  * ... (중략)
  * **루프 9회전 (i=8)**: `i++`에 의해 `i = 9`. 홀수이므로 `continue` 가동.
  * **루프 10회전 (i=9)**: `i < 10` $\rightarrow$ `9 < 10` (참, 진입)
    * `i++`에 의해 `i = 10`.
    * `i % 2`는 `0`이므로 짝수 패스 $\rightarrow$ `sum += 10` 누적. (`sum = 2 + 4 + 6 + 8 + 10 = 30`)
  * **루프 조건 검사 (i=10)**: `i < 10` $\rightarrow$ `10 < 10` (거짓, 루프 종료)
* **최종 결과**:
  * 1부터 10까지의 수 중에서 홀수는 `continue`로 패스하고, 짝수만 `sum`에 누적되어 최종 합계인 **`30`**이 출력됩니다.

## 61. 운영체제 - 메모리 참조 지역성(Locality of Reference)의 3대 유형

### 문제
다음은 지역성에 대한 설명이다. 괄호 ( ) 안에 들어갈 알맞은 유형을 쓰시오.
* 지역성은 크게 ( ① ) 지역성, ( ② ) 지역성, ( ③ ) 지역성이 있다.
* ( ① ) 지역성은 최근 사용되었던 기억장소들이 집중적으로 액세스하는 현상이고,

* ( ③ ) 지역성은 데이터가 순차적으로 액세스 되는 현상이다.

### 정답 및 풀이 결과

* **모범 답안**:
  * **①: 시간 (Temporal)**
  * **②: 공간 (Spatial)**
  * **③: 순차 (Sequential)**
* **결과**: **이론 완벽 정리 완료**

### 핵심 해설
* **참조 지역성(Locality)이란?**:
  * CPU가 주기억장치의 데이터를 참조할 때, 특정 범위나 특정 시간 동안 집중적으로 참조하여 **예측 가능한 패턴**을 보이는 현상입니다. 이 특성을 이용하여 **캐시 메모리(Cache Memory)의 적중률(Hit Ratio)**을 극대화하고 워킹 셋(Working Set)을 유지합니다.
* **3대 참조 지역성 세부 분석**:
  1. **시간 지역성 (Temporal Locality) (지문 ①번)**:
     * *원리*: 최근에 참조된 메모리 주소는 **조만간(가까운 미래에) 또다시 재참조**될 확률이 매우 높다는 특성입니다.
     * *예시 코드*: 반복문(Loop) 내부의 제어 변수(`i`), 집계 변수(`sum`), 공통 서브루틴(함수 호출), 스택(Stack) 포인터.
  2. **공간 지역성 (Spatial Locality) (지문 ②번)**:
     * *원리*: 특정 메모리 위치에 접근했다면, **그 주소와 물리적으로 아주 가까운 인접 주소 영역**을 집중적으로 사용할 확률이 높다는 특성입니다.
     * *예시 코드*: 배열(Array) 데이터 탐색, 공통 선언된 연속적 변수 그룹.
  3. **순차 지역성 (Sequential Locality) (지문 ③번)**:
     * *원리*: 데이터가 **기억장치에 저장된 순서대로 차례차례 액세스**되는 현상입니다. 공간 지역성의 특수한 일종으로 분류되기도 합니다.
     * *예시 코드*: 프로그램의 명령어 순차적 실행 (프로그램 카운터 PC가 1씩 자동 증가하며 다음 주소를 읽어오는 과정).

## 62. C언어 - 구조체 포인터를 이용한 단방향 연결 리스트(Linked List) 멤버 연산

### 문제
아래 C 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 1라인의 `ffinclude <stdio.h)` 등 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```c
#include <stdio.h>
struct node {
    int d; 
    struct node *next;
};
int main() {
    struct node t[3]={
        {1, NULL},
        {2, NULL},
        {3, NULL}
    };
    t[0].next=&t[1];
    t[1].next=&t[2];
    struct node *p=&t[1];
    printf("%d", p->d-1);
    return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `1` (연결 리스트 주소 체인 형성, 포인터 멤버 간접 참조 연산자 및 산술 연산 정확히 분석 완료)
* **모범 답안**: **`1`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **구조체 배열 초기화 및 포인터 연계**:
  * 크기가 3인 구조체 배열 `t`가 순서대로 값(`1, 2, 3`)과 `NULL` 포인터로 생성됩니다.
  * **주소 연결 수립**:
    * `t[0].next = &t[1];` $\rightarrow$ 0번 노드의 `next` 포인터가 1번 노드 `t[1]`의 주소를 가리킵니다.
    * `t[1].next = &t[2];` $\rightarrow$ 1번 노드의 `next` 포인터가 2번 노드 `t[2]`의 주소를 가리킵니다.
    * 최종 체인: `t[0](d=1) -> t[1](d=2) -> t[2](d=3) -> NULL`
* **포인터 지정 및 간접 참조 연산 (`p->d - 1`)**:
  * `struct node *p = &t[1];` 에 의해 포인터 변수 `p`는 `t[1]` 노드의 시작 메모리 주소를 담고 있습니다.
  * **`p->d`**: 포인터가 지목하는 멤버 `d` 값을 가져오므로 `t[1].d` 값인 **`2`**가 반환됩니다.
  * **연산자 우선순위**: 멤버 간접 참조 연산자(`->`)는 뺄셈 연산자(`-`)보다 결합 우선순위가 훨씬 높으므로 `(p->d) - 1` 과 같이 결합됩니다.
  * 계산식: `2 - 1 = 1`이 되어 최종적으로 화면에 **`1`**이 출력됩니다.

## 63. Java - String 클래스 메서드 활용 (replace, concat, substring)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인 끝의 `；` 등 문장 부호 OCR 오타는 정상 코드로 보정하여 풀이한다.)
```java
public class Soojebi {
    public static void main(String[] args) {
        String s="apple";
        s=s.replace('p', 'b');
        s=s.concat("s");
        System.out.println(s.substring(1, 4));
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `bbl` (문자 치환, 문자열 연결 및 서브스트링 인덱스 추출 규칙 완벽 적용)
* **모범 답안**: **`bbl`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **자바 String API 메서드별 상세 연산**:
  1. **`s.replace('p', 'b')`**:
     * 문자열 `"apple"` 안에 존재하는 모든 문자 `'p'`를 `'b'`로 변환합니다.
     * 결과: `"apple"` $\rightarrow$ **`"abble"`**
  2. **`s.concat("s")`**:
     * 변환된 문자열 `"abble"`의 끝에 문자열 `"s"`를 연결합니다.
     * 결과: `"abble" + "s"` $\rightarrow$ **`"abbles"`**
  3. **`s.substring(1, 4)`**:
     * 문자열 `"abbles"`의 인덱스 `1`부터 `4 - 1 = 3`까지의 부분 문자열을 잘라냅니다. (시작 인덱스는 포함하고, 끝 인덱스는 미포함하는 파이썬 슬라이싱과 동일한 규칙)
     * *인덱스 매핑*: `a(0), b(1), b(2), l(3), e(4), s(5)`
     * 결과: `1`번인 `'b'`부터 `3`번인 `'l'`까지인 **`"bbl"`**을 반환합니다.
* **출력**:
  * 최종 반환된 부분 문자열인 **`bbl`**이 화면에 출력됩니다.

## 64. 운영체제 - HRN(Highest Response-ratio Next) 스케줄링 평균 대기 시간 연산

### 문제
프로세스의 도착 시간과 서비스 시간은 다음과 같다. HRN 스케줄링을 적용할 때 평균 대기 시간을 구하시오.

| 프로세스 | 도착 시간 | 서비스 시간 |
| :--- | :--- | :--- |
| **P1** | 0 | 2 |
| **P2** | 1 | 3 |
| **P3** | 2 | 5 |
| **P4** | 3 | 3 |

### 정답 및 풀이 결과
* **작성한 답**: `2.25` (HRN 우선순위 산정 공식 및 타임 흐름별 디버깅 과정을 정확히 계산하여 도출)
* **모범 답안**: **`2.25`**
* **결과**: **정답 (100% 매칭 및 계산 성공)**

### 핵심 해설
* **HRN (Highest Response-ratio Next) 스케줄링 공식**:
  * 비선점형 스케줄링 기법으로, SJF(Shortest Job First)의 약점인 기아 현상(Starvation)을 해결하기 위해 대기 시간과 서비스 시간을 함께 반영하여 우선순위를 매깁니다.
  * **$\text{우선순위 수치} = \frac{\text{대기 시간} + \text{서비스 시간}}{\text{서비스 시간}}$** (수치가 **클수록** 우선순위가 높음)

* **시간대별 프로세스 실행 흐름 상세 분석**:
  1. **$\text{Time} = 0 \sim 2$ (P1 실행)**:
     * 도착 프로세스: **P1**만 유일하게 도착한 상태이므로 바로 실행을 개시합니다.
     * P1 완료 시점: $Time = 2$
     * P1 대기 시간: $2 - 2 - 0 = 0$
  2. **$\text{Time} = 2$ (P2 실행 결정)**:
     * 준비 큐 대기 프로세스:
       * **P2** (도착 $1$, 대기 시간 $= 2 - 1 = 1$)
       * **P3** (도착 $2$, 대기 시간 $= 2 - 2 = 0$)
     * 우선순위 수치 계산:
       * **P2 우선순위**: $\frac{1 + 3}{3} = \frac{4}{3} \approx 1.33$
       * **P3 우선순위**: $\frac{0 + 5}{5} = \frac{5}{5} = 1.00$
     * 판정: P2의 우선순위가 더 높으므로 **P2**가 선택되어 $Time = 2 \sim 5$ 동안 실행됩니다. (P2 최종 대기 시간 $= 1$)
  3. **$\text{Time} = 5$ (P4 실행 결정)**:
     * 준비 큐 대기 프로세스:
       * **P3** (도착 $2$, 대기 시간 $= 5 - 2 = 3$)
       * **P4** (도착 $3$, 대기 시간 $= 5 - 3 = 2$)
     * 우선순위 수치 계산:
       * **P3 우선순위**: $\frac{3 + 5}{5} = \frac{8}{5} = 1.60$
       * **P4 우선순위**: $\frac{2 + 3}{3} = \frac{5}{3} \approx 1.67$
     * 판정: P4의 우선순위가 더 높으므로 **P4**가 선택되어 $Time = 5 \sim 8$ 동안 실행됩니다. (P4 최종 대기 시간 $= 2$)
  4. **$\text{Time} = 8$ (P3 실행 결정)**:
     * 준비 큐에 남은 유일한 프로세스인 **P3**가 $Time = 8 \sim 13$ 동안 최종 실행됩니다.
     * P3 최종 대기 시간: $8 - 2 = 6$

* **평균 대기 시간 연산**:
  * 각 프로세스별 대기 시간: P1 $= 0$, P2 $= 1$, P3 $= 6$, P4 $= 2$
  * 대기 시간의 총합: $0 + 1 + 6 + 2 = 9$
  * **$\text{평균 대기 시간} = \frac{9}{4} = 2.25$**

## 65. 소프트웨어 개발 보안 - 패스워드 크래킹 및 ARP 스푸핑(Spoofing) 공격 기법

### 문제
다음은 보안 공격에 대한 설명이다. 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* 시스템 또는 서비스의 ID와 패스워드를 크랙하기 위해서 ID와 패스워드가 될 가능성이 있는 단어를 파일로 만들어 놓고 이 파일의 단어를 대입하여 크랙하는 공격 기법인 ( ① ) 공격 기법
* 패스워드로 사용될 수 있는 영문자(대소문자), 숫자, 특수문자 등을 무작위로 패스워드 자리에 대입하여 패스워드를 알아내는 공격기법인 ( ② ) 공격 기법
* 공격자가 특정 호스트의 MAC 주소를 자신의 MAC 주소로 위조한 ARP Reply를 만들어 희생자에게 지속적으로 전송하여 희생자의 ARP Cache Table에 특정 호스트의 MAC 정보를 공격자의 MAC 정보로 변경, 희생자로부터 특정 호스트로 나가는 패킷을 공격자가 스니핑하는 기법인 ( ③ ) 공격 기법

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 사전 공격** (단어 파일을 사전처럼 사용함을 정확히 유추해 도출 가능)
  * **②: 브루트 포스** (무작위 대입의 동의어를 완벽히 정답 처리)
  * **③: ARP 스니핑** (스니핑 행위는 맞으나, 공격 기법 자체의 명칭인 '스푸핑'과의 미세 명명 차이 발생)
* **모범 답안**:
  * **①: 사전 공격 (Dictionary Attack)**
  * **②: 무차별 대입 공격 (또는 `무작위 대입 공격`, `브루트 포스 공격`, `Brute-Force Attack`)**
  * **③: ARP 스푸핑 (ARP Spoofing)** (또는 `ARP 스푸핑 공격`)
* **결과**: **부분 수정 후 최종 정답 도출 가능**

### 핵심 해설
* **패스워드 크래킹 2대 기법 (①, ②번)**:
  * **사전 공격 (Dictionary Attack)**: 사전에 자주 쓰이는 단어, 인명, 숫자 조합 등을 단어 파일(Dictionary File)로 구축한 뒤 하나씩 꺼내 대입해 보는 기법입니다.
  * **무차별 대입 공격 (Brute-Force Attack)**: 가능한 모든 영문자, 숫자, 특수문자 조합을 무작위로 전부 대입하여 알아내는 무식하지만 확실한 해킹 기법입니다.
* **네트워크 위조 공격과 도청 행위의 명확한 용어 구분 (③번 - 중요 함정)**:
  * **스푸핑 (Spoofing - 속이기)**: 승인된 호스트나 도메인 등으로 **자신의 정체를 속여(위조)** 시스템에 우회 접근하는 기법 자체를 뜻합니다. (예: ARP 스푸핑, IP 스푸핑 등)
  * **스니핑 (Sniffing - 훔쳐보기/코를 킁킁대다)**: 네트워크 패킷을 몰래 훔쳐보는 **도청 행위**를 뜻합니다.
  * *결론*: 본 문제 지문은 "MAC 주소를 위조한 ARP Reply를 보내 ARP Cache Table을 변조"하는 **공격 기법 자체의 이름**을 묻고 있으므로, 올바른 DDL 명칭은 **`ARP 스푸핑`**입니다. (그를 통해 패킷을 스니핑하게 됩니다.)
