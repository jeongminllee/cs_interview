## 146. 운영체제 - 응용 소프트웨어 기초 기술 (프로세스 선점형 스케줄링 라운드 로빈 Round Robin 특징)

### 문제
모든 프로세스에 대해 같은 크기의 CPU 시간을 할당(시간 할당량)하고, 프로세스가 할당된 시간 내에 처리 완료를 못하면 준비 큐 리스트의 가장 뒤로 보내지고, CPU는 대기 중인 다음 프로세스로 넘어가는 스케줄링 기법은 무엇인지 쓰시오.

### 정답 및 풀이 결과
* **작성한 답**: `라운드 로빈` (동일한 Time Quantum 할당 및 시분할 기반 선점 처리 스케줄링의 고유 정의를 정확하게 식별하여 정답 도출)
* **모범 답안**: **`라운드 로빈 (Round Robin)`** (또는 `Round Robin`, `RR`, `라운드 로빈`)
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 운영체제(OS)의 프로세스 관리 중 CPU를 효율적으로 분배하기 위한 **프로세스 스케줄링 알고리즘 중 시분할 시스템의 대표적인 선점(Preemptive) 스케줄링인 라운드 로빈(Round Robin)의 고유 개념**을 식별하는 문제입니다.

* **라운드 로빈(Round Robin)의 핵심 기출 특징 정밀 분석**:
  1. **동일한 시간 할당량 (Time Slice / Time Quantum)**: 각 프로세스는 규정된 일정 시간 동안만 CPU를 점유할 수 있습니다.
  2. **선점(Preemptive) 방식**: 할당된 시간이 경과하면 CPU 소유권을 강제로 빼앗기고 준비 큐(Ready Queue)의 맨 뒤로 이동합니다.
  3. **시간 할당량 크기에 따른 양면성 (시험 단골 족보)**:
     * *시간 할당량이 너무 크면*: 먼저 들어온 프로세스가 끝날 때까지 대기하게 되므로 **FCFS (First-Come, First-Served) 알고리즘과 동일**해집니다.
     * *시간 할당량이 너무 작으면*: 프로세스 간 빈번한 잦은 전환으로 인해 **문맥 교환(Context Switching) 오버헤드**가 극도로 증가하여 시스템 성능이 저하됩니다.

## 147. 운영체제 - 응용 소프트웨어 기초 기술 (운영체제 OS 2대 핵심 구성 요소 명령어 해석기 쉘 Shell vs 시스템 핵심 커널 Kernel)

### 문제
다음 운영체제 주요 구성 요소에 대한 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① )은/는 사용자가 입력시킨 명령어 라인을 읽어 필요한 시스템 기능을 실행시키는 명령어 해석기로, 시스템과 사용자 간의 인터페이스를 제공한다.
* ( ② )은/는 운영체제의 핵심이 되는 기능들이 모여 있는 컴퓨터 프로그램이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 쉘 (Shell)**
  * **②: 커널 (Kernel)**
* **모범 답안**:
  * **①: 쉘 (Shell)** (또는 `Shell`, `쉘`)
  * **②: 커널 (Kernel)** (또는 `Kernel`, `커널`)
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 컴퓨터 시스템 하드웨어와 사용자 사이를 매개하는 **운영체제(OS)의 가장 기본적인 2대 핵심 구성 요소인 쉘(Shell)과 커널(Kernel)의 고유 기능 정의**를 상호 구별할 수 있는지 검증하는 단답형 기출 문제입니다.

* **운영체제 2대 핵심 요소 정밀 분석**:
  1. **쉘 (Shell) (지문 ①번)**:
     * *역할*: 사용자와 시스템(커널) 사이에서 교량 역할을 수행하는 **명령어 해석기(Command Interpreter)**입니다.
     * *동작*: 사용자가 쉘 프롬프트에서 입력한 CLI 명령어나 스크립트를 해석하여 하위 커널이 이해할 수 있는 시스템 콜(System Call) 형태로 번역해 전달합니다.
     * *대표 종류*: `bash` (리눅스 표준), `sh` (Bourne 쉘), `csh` (C 쉘), `zsh` 등.
  2. **커널 (Kernel) (지문 ②번)**:
     * *역할*: 운영체제의 최심부 심장이 되는 **핵심 제어 프로그램**입니다.
     * *동작*: 부팅 직후 주기억장치(RAM)에 상주하면서 하드웨어 자원(CPU, 메모리, 디스크, I/O 장치 등)을 직접 제어하고 관리합니다.
     * *5대 주요 관리 기능*: 프로세스 관리, 기억장치(메모리) 관리, 파일 시스템 관리, 입출력(I/O) 장치 관리, 프로세스 간 통신(IPC) 및 보안.

## 148. SQL 응용 - 데이터베이스 구축 (교수 테이블 조회 SQL 작성 ORDER BY 절의 다중 컬럼 내림차순 DESC 및 오름차순 ASC 적용)

### 문제
[교수] 테이블에서 다음 조건에 부합하도록 데이터를 조회하는 SQL 구문을 작성하시오. (단, 마크다운 HTML 코드 블록에서 문제 조건과 테이블 구조를 정밀 추출하여 풀이한다.)
* **[조건]**: 강의실은 내림차순(DESC)으로 정렬한 후, 같은 값일 때 이름을 오름차순(ASC)으로 정렬하시오.
* **[교수 테이블]**: (컬럼: `이름`, `과목`, `강의실`)

### 정답 및 풀이 결과
* **작성한 답**: `SELECT * FROM 교수 ORDER BY 강의실 DESC, 이름 ASC;` (오름차순 키워드 ASC의 명시적 작성 여부에 대한 채점 안전성을 고려하여 무결한 1·2차 다중 정렬 구문 작성 성공)
* **모범 답안**: **`SELECT * FROM 교수 ORDER BY 강의실 DESC, 이름 ASC;`** (또는 `SELECT * FROM 교수 ORDER BY 강의실 DESC, 이름;`)
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 관계형 데이터베이스(RDBMS)의 표준 SQL 구문 중 **`SELECT` 문의 `ORDER BY` 절을 활용하여 다중 컬럼(Multi-Column)을 기준으로 1차, 2차 정렬 순서와 정렬 방향(오름차순 `ASC` vs 내림차순 `DESC`)을 지정**할 수 있는지 검증하는 빈출 DML 작성 문제입니다.

* **SQL ORDER BY 구문 정밀 해체**:
  1. **기본 조회 절**: `SELECT * FROM 교수` $\rightarrow$ 교수 테이블의 모든 속성(`이름`, `과목`, `강의실`)을 가져옵니다.
  2. **1차 정렬 조건 (`강의실 DESC`)**:
     * `강의실` 컬럼을 기준으로 **내림차순(DESC, Descending)** 정렬합니다.
     * 알파벳 순서상 `'B'`가 `'A'`보다 크므로, `'B'` 강의실인 수제비쌤 행이 최상단(1행)에 먼저 배치됩니다.
  3. **2차 정렬 조건 (`이름 ASC`)**:
     * 1차 정렬 결과에서 `강의실` 값이 동일하게 `'A'`로 일치하는 행들(두음쌤, 보안쌤, 지기쌤)에 한해 **2차 정렬**을 적용합니다.
     * `이름` 컬럼을 기준으로 **오름차순(ASC, Ascending)** 정렬합니다.
     * 한글 가나다순으로 `'두음쌤'` ➔ `'보안쌤'` ➔ `'지기쌤'` 순서로 행이 정렬됩니다.
  4. **`ASC` 키워드 작성 팁**:
     * SQL에서 `ORDER BY` 시 정렬 키워드를 생략하면 기본값(Default)으로 `ASC`(오름차순)가 적용됩니다.
     * 하지만 정보처리기사 실기 시험지 조건문에 **"오름차순으로 정렬"**이 명시되어 있을 때에는, 채점위원의 명시성 감점 요인을 사전에 차단하기 위해 **`ASC` 키워드를 명시적으로 꼭 표기**해 주는 것이 가장 이상적인 정답 작성 방식입니다!

## 149. Python - 프로그래밍 언어 활용 (딕셔너리 Dictionary get 메서드의 존재하지 않는 키 참조 및 기본값 Default Value 반환)

### 문제
아래 파이썬 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 코드 블록 HTML 내에서 바르게 파이썬 코드로 디해싱하여 풀이한다.)
```python
a = {'name': 'soojebi', 'phone': '010-1234-5678', 'birth': '0909'}
print(a.get('age', 'none'))
```

### 정답 및 풀이 결과
* **작성한 답**: `none` (HTML 토큰에서 파이썬 딕셔너리 구문을 정확히 복원하고, get 메서드가 부재 키 참조 시 2번째 인자인 기본값을 반환하는 메커니즘을 적용하여 정답 도출 성공)
* **모범 답안**: **`none`**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 파이썬의 핵심 자료구조인 **딕셔너리(Dictionary) 객체의 `get()` 메서드 동작 원리**와, 존재하지 않는 키(Key)를 참조할 때 **오류(KeyError) 방지를 위한 기본값(Default Value) 설정 기능**을 바르게 이해하고 있는지 검증하는 단답형 계산 문제입니다.

* **파이썬 딕셔너리 `get(key, default)` 동작 정밀 해체**:
  1. 딕셔너리 `a`에는 `'name'`, `'phone'`, `'birth'` 세 개의 키(Key)가 존재합니다.
  2. `a.get('age', 'none')` 구문이 실행되면, 먼저 첫 번째 매개변수인 `'age'` 키가 딕셔너리 `a`에 존재하는지 조회합니다.
  3. **키 참조 및 예외 방어 기작**:
     * `'age'` 키는 딕셔너리 `a`에 존재하지 않습니다.
     * 일반 대괄호 참조 방식인 `a['age']`로 접근하면 `KeyError` 예외가 발생하며 프로그램이 즉시 중단되지만, **`get()` 메서드를 사용하면 예외가 터지지 않습니다.**
     * 두 번째 인자로 지정된 기본값 문자열인 **`'none'`** 이 최종 반환됩니다.
  4. `print()` 함수는 이 문자열의 따옴표를 떼어내고 화면에 **`none`** 을 출력합니다.
  5. **★ [수험생 실전 통찰 팁] 딕셔너리 카운팅(Counter) 구현 패턴**:
     * 실전 프로그래밍에서 빈도수(Frequency)를 집계할 때 `dict[key] = dict.get(key, 0) + 1` 공식을 사용하는 이유가 바로 이 지점에 있습니다.
     * 키가 최초 등장할 때는 `get(key, 0)`에 의해 기본값 `0`이 반환된 뒤 `+ 1`이 되어 `1`로 초기 적재되고, 이미 존재하는 키일 때는 기존 카운트 값에 `+ 1`이 중첩되어 `KeyError` 없이 안전하게 카운터(Counter) 로직을 완성하게 됩니다.

## 150. Java - 프로그래밍 언어 활용 (클래스 로딩 정적 static 블록 인스턴스 초기화 블록 부모 생성자 super 및 오버로딩 순서 정밀 트레이싱)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 코드 블록 HTML 내에서 바르게 자바 코드로 디해싱하여 풀이한다.)
```java
class Super {
    static int a = 10;
    int b = 20;
    static { a += 30; }
    { a += 5; b += 5; }

    Super() { b += 5; }

    void display() { System.out.print(a + b + " "); }
}
class Sub extends Super {
    static { a += 10; }
    { a += 10; b += 10; }
    Sub() { b += 5; }
    Sub(int n) { b += n; }
    @Override
    void display() { System.out.print(a + b + " "); }
}
public class Soojebi {
    public static void main(String[] args) {
        Super obj1 = new Super();
        obj1.display();
        Super obj2 = new Sub();
        obj2.display();
        Sub obj3 = new Sub(30);
        obj3.display();
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `75 115 155 ` (정적 변수 a의 공유 누적과 인스턴스 변수 b의 독립 초기화, 부모-자식 블록 가동 순서를 정밀 추적하여 킬러 문항 정답 도출 성공)
* **모범 답안**: **`75 115 155 `** (또는 `75 115 155`)
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 자바 프로그래밍 시험에서 최고 난이도로 출제되는 **JVM의 클래스 로딩 시점(`static { }`), 인스턴스 초기화 블록(`{ }`), 부모 생성자(`super()`) 자동 호출 순서, 그리고 정적 변수(`static`)와 인스턴스 변수 간의 생명주기 차이**를 정밀하게 트레이싱할 수 있는지 검증하는 최상급 융합 문제입니다.

* **JVM 4단계 객체 생성 및 초기화 우선순위 족보**:
  1. **`1순위: 클래스 로딩 (static)`** $\rightarrow$ JVM에 해당 클래스가 처음 참조될 때 딱 1번만 실행 (`static` 변수 및 `static { }` 블록 가동).
  2. **`2순위: 부모 클래스의 인스턴스 초기화`** $\rightarrow$ 자식 객체 생성 시 부모의 `int b = 20;` ➔ 부모 인스턴스 블록 `{ }` ➔ 부모 생성자 `Super()` 순서 가동.
  3. **`3순위: 자식 클래스의 인스턴스 초기화`** $\rightarrow$ 자식 인스턴스 블록 `{ }` 가동.
  4. **`4순위: 자식 생성자 본문 실행`** $\rightarrow$ 호출된 자식 생성자 `Sub()` 또는 `Sub(int n)` 실행.

* **메인 메서드 3개 객체 생성 트레이싱 분석**:

  1. **`Super obj1 = new Super();`**:
     * *`Super` 클래스 로딩*: `static int a = 10;` ➔ `static { a += 30; }` $\rightarrow$ 정적 변수 **$a = 40$**
     * *`obj1` 인스턴스 생성*: `b = 20` ➔ `Super` 인스턴스 블록 `{ a += 5; b += 5; }` ($a=45, b=25$) ➔ `Super()` 생성자 (`b += 5` $\rightarrow$ $b=30$)
     * *`obj1.display()`*: $a + b = 45 + 30 =$ **`75 `** 인쇄 ($a$는 공유 변수이므로 45 유지)

  2. **`Super obj2 = new Sub();`**:
     * *`Sub` 클래스 최초 로딩*: `Sub` 정적 블록 `static { a += 10; }` 가동 $\rightarrow$ **$a = 45 + 10 = 55$**
     * *`obj2` 인스턴스 생성*:
       * ① **부모 `Super` 가동**: `b = 20` ➔ `Super` 인스턴스 블록 ($a=60, b=25$) ➔ `Super()` 생성자 ($b=30$)
       * ② **자식 `Sub` 가동**: `Sub` 인스턴스 블록 `{ a += 10; b += 10; }` ($a=70, b=40$) ➔ `Sub()` 생성자 (`b += 5` $\rightarrow$ $b=45$)
     * *`obj2.display()`*: $a + b = 70 + 45 =$ **`115 `** 인쇄 ($a$는 70으로 갱신)

  3. **`Sub obj3 = new Sub(30);` (★ 수험생의 $b$ 변수 미스터리 완벽 해소)**:
     * *클래스 로딩*: `Super`, `Sub` 모두 이미 완료되었으므로 `static` 블록은 실행되지 않음! (출발점 $a = 70$)
     * *`obj3` 인스턴스 생성*:
       * ① **부모 `Super` 가동**: `b = 20` ➔ `Super` 인스턴스 블록 ($a=75, b=25$) ➔ `Super()` 생성자 ($b=30$)
       * ② **자식 `Sub` 가동**:
         * 자식 인스턴스 블록 `{ a += 10; b += 10; }` 이 생성자 본문보다 **먼저** 가동됩니다!
         * 이로 인해 $a = 75 + 10 = 85$, **$b = 30 + 10 = 40$** 가 됩니다!
         * 마지막으로 호출된 오버로딩 생성자 `Sub(30)` 본문 `b += 30` 이 실행되어 **$b = 40 + 30 = 70$** 이 완성됩니다!
     * *`obj3.display()`*: $a + b = 85 + 70 =$ **`155 `** 인쇄!

  4. **★ [수험생과의 질답 Q&A 심화 필기 노트] Java 상속 생성자 super() 매칭 메커니즘**:
     * **Q1. 자식 생성자 `Sub(30)`을 호출했을 때, 명시하지 않은 부모 생성자 `Super()`는 왜 가동하는가?**
       * **A1**: 자바 문법 규칙상 자식 클래스의 모든 생성자 맨 첫 줄에는 컴파일러가 생략된 **`super();`** 문장을 자동으로 삽입합니다. 이는 부모의 뼈대(인스턴스 변수 및 초기화 블록)가 먼저 완전하게 메모리에 구축되어야 자식이 안전하게 상속받아 사용할 수 있다는 자바 객체지향의 철칙 때문입니다. (아파트 뼈대 공사 후 내부 인테리어 착수 비유)
     * **Q2. `super()` 와 `super(args)` 의 매칭 원리는?**
       * **A2**:
         1. 개발자가 아무것도 안 적거나 `super()` 를 명시한 경우 $\rightarrow$ 부모 클래스의 **인자 없는 기본 생성자 `Super()`** 매칭 및 호출.
         2. 개발자가 `super(object)` 처럼 인자를 명시한 경우 $\rightarrow$ 부모 클래스의 **해당 인자 타입을 매개변수로 받는 오버로딩 생성자 `Super(object)`** 가 명시적 매칭 및 호출.

## 151. Java - 프로그래밍 언어 활용 (예외 처리 try-catch-finally 블록 내 return 문의 finally 우선 가동 및 static 변수 연산 출력)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 코드 블록 HTML 내에서 바르게 자바 코드로 디해싱하여 풀이한다.)
```java
public class Soojebi {
  static int count = 0;
  public static void main(String[] args) {
    int res = test(10);
    System.out.println(res + " " + count);
  }
  public static int test(int n) {
    try {
      count++;
      if (n > 5) {
          n += count;
          return n;
      }
      return n;
    }
    catch (Exception e) {
        count -= 1;
        return 0;
    }
    finally {
      count += 10;
      n += 100;
      System.out.print("F ");
    }
  }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `11 1` (try 문 내의 return 구문 만남 시 메서드가 조기 종료된다고 오해하여, finally 블록의 무조건 실행 원칙과 내부 System.out.print("F ") 출력 누락으로 정답 도출 실패)
* **모범 답안**: **`F 11 11`**
* **결과**: **오답 (finally 블록의 return 선행 가동 및 내부 print 출력 누락)**

### 핵심 해설
이 문제는 자바 프로그래밍의 예외 처리(Exception Handling) 구조인 **`try-catch-finally` 문에서 `try` 블록 내부의 `return` 문을 만나더라도, 메서드가 즉시 탈출하지 않고 `finally` 블록을 100% 선행 가동시킨 후 최종 반환하는 자바의 필수 보장 매커니즘**을 이해하고 있는지 검증하는 킬러 함정 문제입니다.

* **단계별 트레이싱 및 2대 초빈출 킬러 함정 해체**:
  1. **`test(10)` 메서드 호출 및 `try` 블록 진입**:
     * `count++;` $\rightarrow$ 정적 변수 `count` 값이 0에서 **1**로 증가합니다.
     * `if (n > 5)` $\rightarrow$ $10 > 5$ 는 **True (참)** 입니다.
     * `n += count;` $\rightarrow$ $n = 10 + 1 =$ **`11`** 이 됩니다.
     * `return n;` $\rightarrow$ 반환값 `11`을 임시 보관(예약)하지만, **메서드를 바로 탈출하지 못하고 `finally` 블록으로 강제 이동합니다.**
  2. **`finally` 블록 무조건 가동 (★ 킬러 함정 구간)**:
     * `count += 10;` $\rightarrow$ `count` 값이 $1 + 10 =$ **`11`** 로 갱신됩니다!
     * `n += 100;` $\rightarrow$ $n$이 111이 되지만, 이미 `return`할 반환값(11)은 `try`문에서 11로 임시 확정되었으므로 리턴값에는 영향을 주지 않습니다.
     * **`System.out.print("F ");`** 가 가동되어 화면에 **`F `** 가 즉시 1차 인쇄됩니다.
  3. **메서드 탈출 및 `main` 복귀**:
     * `test(10)` 메서드는 임시 보관해둔 반환값 **`11`** 을 리턴하며 최종 종료됩니다.
     * `res` = 11, `count` = 11
  4. **`main` 메서드 출력문 실행**:
     * `System.out.println(res + " " + count);` 가 가동되어 **`11 11`** 이 2차 인쇄됩니다.
  5. **최종 종합 출력물**:
     * 1차 인쇄된 `F ` 뒤에 2차 인쇄된 `11 11` 이 이어지므로 최종 결과는 **`F 11 11`** 이 됩니다.
  6. **★ [수험생의 핵심 통찰 노하우] return 선행과 static 전역 변수의 관계**:
     * `try` 문 내부에서 `return n;` 을 만나는 순간 기본 타입 리턴값(`11`)은 이미 확정되어 보관되며, 그 즉시 `finally` 블록으로 넘어가 선행 실행됩니다.
     * `finally` 내부에서 정적 변수(`static count`)는 파이썬의 `global` 전역 변수처럼 프로그램 전체에 연산이 그대로 즉시 반영되어 `count = 11`이 되고, internal `print("F ")` 가 1차 인쇄된 후 탈출하여 최종 `F 11 11` 구조를 완성하게 됩니다.

## 152. Java - 프로그래밍 언어 활용 (this 키워드 누락에 따른 매개변수 섀도잉 Variable Shadowing 무효화 및 static 객체 공유 변수 연산)

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 수험생이 기술한 복원 코드에 기초하여 정밀 풀이한다.)
```java
class Person {
    String name;
    static int cnt = 0;

    Person(String name) {
        this.name = name;
        cnt++;
    }
    Person() {
        this.name = null;
        cnt++;
    }
    void setName(String name) {
        name = name; // ★ this 키워드 누락 킬러 함정 구간!
    }
    String getName() {
        return "Person";
    }
}

public class Soojebi {
    public static void main(String[] args) {
        Person p1 = new Person("soojebi");
        Person p2 = new Person();
        p1.setName("java");
        System.out.println(p1.name);
        System.out.println(Person.cnt);
        System.out.println(p1.getName());
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**:
  `soojebi`
  `2`
  `Person` (this 키워드 누락 시 인스턴스 멤버 변수가 갱신되지 않는 변수 섀도잉 현상을 정밀 간파하고 static cnt의 객체 간 공유 특징을 적용하여 정답 도출 성공)
* **모범 답안**:
  **`soojebi`**
  **`2`**
  **`Person`**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 자바의 클래스 설계 시 멤버 변수(Field)와 메서드 매개변수(Parameter)의 이름이 동일할 때, **`this` 참조 키워드의 유무에 따른 변수 섀도잉(Variable Shadowing) 현상**과, 모든 인스턴스가 단 하나의 메모리 공간을 공유하는 **정적 변수(`static cnt`)의 누적 동작**을 정밀하게 트레이싱할 수 있는지 검증하는 킬러 문제입니다.

* **단계별 트레이싱 및 킬러 함정 해체**:
  1. **`Person p1 = new Person("soojebi");`**:
     * `p1.name = "soojebi"` 로 설정됩니다.
     * `cnt++` $\rightarrow$ 정적 변수 `cnt`가 0에서 **1**로 증가합니다.
  2. **`Person p2 = new Person();`**:
     * `p2.name = null` 로 설정됩니다.
     * `cnt++` $\rightarrow$ 정적 변수 `cnt`는 모든 객체가 공유하므로 $1 + 1 =$ **`2`** 가 됩니다.
  3. **`p1.setName("java");` (★ 초빈출 킬러 함정 구간)**:
     * `setName(String name)` 메서드 내부 코드가 `this.name = name;` 이 아니라 **`name = name;`** 으로 작성되어 있습니다.
     * 이 구문은 인스턴스 필드인 `this.name`을 수정하는 것이 아니라, **자기 자신인 지역 매개변수 `name`에 자기 자신("java")을 다시 대입하고 끝나는 자기할당(Self-Assignment) 무효 연산**입니다.
     * 따라서 인스턴스 변수 `p1.name` 은 전혀 변경되지 않고 그대로 **`"soojebi"`** 로 유지됩니다!
  4. **출력문 3줄 가동**:
     * `System.out.println(p1.name);` ➔ **`soojebi`**
     * `System.out.println(Person.cnt);` ➔ **`2`**
     * `System.out.println(p1.getName());` ➔ 메서드가 리턴하는 문자열 **`Person`**

## 153. C언어 - 프로그래밍 언어 활용 (문자열 포인터 증감 연산자 p++ 및 ++p와 서식 지정자 %s %c 동작 트레이싱)

### 문제
아래 C언어 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 수험생이 기술한 복원 코드 및 풀이 조건에 기초하여 정밀 해설한다.)
```c
#include <stdio.h>
int main() {
    char *p = "soojebi world";
    printf("%s\n", p++);
    printf("%s\n", ++p);
    printf("%c\n", *p);
    printf("%c\n", *p++);
    return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**:
  `soojebi world`
  `ojebi world`
  `o`
  `o` (포인터 전위/후위 증감연산의 실행 시점과 서식자 %s 및 %c의 데이터 읽기 단위를 정확하게 적용하여 정답 도출 성공)
* **모범 답안**:
  **`soojebi world`**
  **`ojebi world`**
  **`o`**
  **`o`**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 C언어의 핵심인 **문자열 포인터(Pointer)의 증감 연산자(`p++` vs `++p`)에 따른 주소 이동 시점**과, **서식 지정자 `%s`(문자열) 및 `%c`(단일 문자)의 데이터 참조 방식**을 정확하게 트레이싱할 수 있는지 검증하는 빈출 문제입니다.

* **단계별 라인 트레이싱 분석**:
  1. **`char *p = "soojebi world";`**:
     * 포인터 변수 `p`가 문자열 상수 `"soojebi world"`의 시작 위치인 인덱스 0번 문자 `'s'`의 메모리 주소를 가리킵니다.
  2. **`printf("%s\n", p++);` (1번째 출력)**:
     * **후위 증감(`p++`)**이므로, 현재 `p`가 가리키는 위치(`'s'`)부터 `%s`로 전체 문자열을 인쇄한 후 `p`를 1 증가시킵니다.
     * 인쇄: **`soojebi world`**
     * 주소 이동: `p`는 인덱스 1번인 문자 `'o'`의 주소를 가리키게 됩니다.
  3. **`printf("%s\n", ++p);` (2번째 출력)**:
     * **전위 증감(`++p`)**이므로, 먼저 `p`를 1 증가시킨 후 출력합니다.
     * 주소 이동: `p`는 인덱스 2번인 문자 `'o'`의 주소를 가리킵니다.
     * 인쇄: `%s`는 인덱스 2번 `'o'`부터 문자열 끝까지 인쇄하므로 **`ojebi world`** 가 출력됩니다.
  4. **`printf("%c\n", *p);` (3번째 출력)**:
     * 현재 `p`는 인덱스 2번 문자 `'o'`를 가리키고 있습니다.
     * `%c`는 단일 문자를 인쇄하므로 `*p`에 해당하는 문자 **`o`** 가 1자 인쇄됩니다. (주소 이동 없음)
  5. **`printf("%c\n", *p++);` (4번째 출력)**:
     * `*p++` 연산자 우선순위는 포인터 간접 참조 `*` 와 후위 증감 `++` 의 조합입니다.
     * 현재 `p`가 가리키는 인덱스 2번 문자 **`o`** 를 먼저 `%c`로 1자 인쇄합니다.
     * 인쇄 직후 후위 증감 `p++`가 실행되어 `p`는 인덱스 3번 문자 `'j'`의 주소로 이동합니다.

## 154. C언어 - 프로그래밍 언어 활용 (문자 배열 아스키코드 연산 x[i]-'A' 및 for 조건식 대입 연산 0 거짓 종결 트레이싱)

### 문제
아래 C언어 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 코드 블록 HTML 내에서 바르게 C언어 코드로 디해싱하여 풀이한다.)
```c
#include <stdio.h>

int main(){
  char x[] = "EATS";
  char s[100] = "";
  int i=0;

  for(i=0; i<100; i++)
    s[i] = i%5;

  for(i=0; s[x[i]] += s[x[i]]; i++)
    printf("%d", x[x[i]-'A']);

  return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `0` (i=0 일 때 NULL 문자 '\0'의 정수값 0이 출력되고, i=1 일 때 s['A']의 값 0이 조건식이 되어 for 문이 조기 거짓 종결되는 메커니즘을 정확히 추적 성공)
* **모범 답안**: **`0`**
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 C언어의 **문자 배열 아스키코드 연산(`'A'` = 65, `'E'` = 69)**과, **2차 중첩 배열 인덱스 참조(`x[x[i]-'A']`)**, 그리고 **`for` 문 조건식에 대입 연산자(`+=`)가 사용되었을 때 결과가 `0`이 되면 거짓(False)으로 즉시 종결되는 제어 흐름**을 정밀하게 분석할 수 있는지 평가하는 고난도 킬러 문제입니다.

* **단계별 트레이싱 및 정밀 해체**:
  1. **첫 번째 `for` 루프 (`s` 배열 초기화)**:
     * `s[i] = i % 5` 식이 0부터 99까지 실행됩니다.
     * 인덱스가 5의 배수(`0, 5, 10, ..., 65, 70, 75, ...`)인 위치의 `s[i]` 값은 모두 **`0`** 입니다!
     * 특히, `'A'`(65) 위치의 값: **`s[65] = 65 % 5 = 0`**
     * `'E'`(69) 위치의 값: **`s[69] = 69 % 5 = 4`**
  2. **두 번째 `for` 루프 1회차 (`i = 0`)**:
     * `x[0]` = `'E'` (아스키코드 69)
     * *조건식 평가*: `s[x[0]] += s[x[0]]` $\rightarrow$ `s[69] += s[69]` $\rightarrow$ $4 + 4 =$ **`8`** (0이 아니므로 **True**)
     * *출력문 실행*: `printf("%d", x[x[0]-'A']);`
       * `x[0] - 'A'` = $69 - 65 =$ **`4`**
       * `x[4]` 참조: `"EATS"` 문자열의 4번 인덱스에는 NULL 문자 **`'\0'`** (아스키 0)가 위치함!
       * `printf("%d", '\0')` 에 의해 숫자 **`0`** 가 1차 출력됩니다.
  3. **두 번째 `for` 루프 2회차 (`i = 1`)**:
     * `x[1]` = `'A'` (아스키코드 65)
     * *조건식 평가*: `s[x[1]] += s[x[1]]` $\rightarrow$ `s[65] += s[65]`
     * 1단계에서 `s[65]` 는 **`0`** 이었습니다.
     * 따라서 `s[65] += s[65]` 은 $0 + 0 =$ **`0`** 이 됩니다!
     * **C언어 조건식에서 `0` 은 거짓(False)이므로 `for` 반복문이 즉시 종결(Exit)됩니다!**
  4. **최종 출력**: `i=0` 일 때 출력된 **`0`** 이 최종 정답이 됩니다.
  5. **★ [수험생의 본질 통찰 노트] 필수 아스키 3대 기준값('A'=65)의 위력**:
     * 본 문제는 결국 대문자 시작 기준값인 **`'A' = 65`** 를 정확히 숙지하고 있어야 $65 \% 5 = 0$ 이라는 거짓(False) 종결 스위치를 당길 수 있는 구조였습니다.
     * C언어 실기 시험 대비 필수 3대 아스키 기준값인 **`'A'=65`**, **`'a'=97`**, **`'0'=48`** 은 단답형 및 트레이싱 문제의 생사줄임을 재확인합니다.

## 155. C언어 - 프로그래밍 언어 활용 (이중 논리 부정 연산자 !(!a)의 정수 1 환산 및 for 증감식 i *= a+b 연산자 우선순위 트레이싱)

### 문제
아래 C언어 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 코드 블록 HTML 내에서 바르게 C언어 코드로 디해싱하여 풀이한다.)
```c
#include <stdio.h>

int main(){
  int a = 3;
  int b = !(!a);
  int i = 0;
  
  for(i=1; i<20; i *= a+b)
    printf("%d\n", i);

  return 0;
}
```

### 정답 및 풀이 결과
* **작성한 답**: `1\n6` (이중 논리 부정 !(!a)의 연산 결과가 원본 값 3이 된다고 착각하여 b=3 및 a+b=6 으로 판단해 정답 도출 실패)
* **모범 답안**:
  **`1`**
  **`4`**
  **`16`** (또는 `1\n4\n16`)
* **결과**: **오답 (이중 논리 부정의 C언어 정수 1 환산 특징 및 복합 대입 연산자 곱셈 누적 오류)**

### 핵심 해설
이 문제는 C언어의 **논리 부정 연산자(`!`)가 0이 아닌 참(True) 값에 적용될 때 정수 `1` 또는 `0`으로 변환되는 특징**과, **`for` 문 증감식의 복합 대입 연산자(`*=`)와 산술 연산자(`+`) 간의 우선순위 연산**을 정확하게 계산할 수 있는지 검증하는 빈출 함정 문제입니다.

* **단계별 트레이싱 및 킬러 함정 해체**:
  1. **`int b = !(!a);` (★ 초빈출 킬러 함정 구간)**:
     * `a = 3` (C언어에서 0이 아닌 정수 3은 **참 True** 입니다.)
     * 1차 부정 `!a` $\rightarrow$ `!3` $\rightarrow$ 참을 부정하므로 정수 **`0` (거짓 False)** 이 됩니다.
     * 2차 부정 `!(!a)` $\rightarrow$ `!0` $\rightarrow$ 거짓을 부정하므로 정수 **`1` (참 True)** 이 됩니다!
     * **자바나 파이썬처럼 원본 값 3으로 돌아가는 것이 아니라, C언어의 논리 연산 결과는 무조건 정수 `1`로 치환되어 `b = 1` 이 됩니다!**
  2. **`for(i=1; i<20; i *= a+b)` 증감식 구조 해석**:
     * `a + b` = $3 + 1 =$ **`4`**
     * `i *= a + b` 식은 대입 연산자 우선순위에 의해 **`i = i * (a + b)`** 로 묶입니다.
     * 즉, `i` 값에 매 루프마다 **`4` 를 곱하는 연산**(`i = i * 4`)입니다.
  3. **`for` 루프 반복 수행**:
     * **1회차 (`i = 1`)**: $1 < 20$ (참) $\rightarrow$ `printf`에 의해 **`1`** 인쇄.
       * 증감식: `i` = $1 \times 4 =$ **`4`**
     * **2회차 (`i = 4`)**: $4 < 20$ (참) $\rightarrow$ `printf`에 의해 **`4`** 인쇄.
       * 증감식: `i` = $4 \times 4 =$ **`16`**
     * **3회차 (`i = 16`)**: $16 < 20$ (참) $\rightarrow$ `printf`에 의해 **`16`** 인쇄.
       * 증감식: `i` = $16 \times 4 =$ **`64`**
     * **4회차 (`i = 64`)**: $64 < 20$ (**거짓 False**) $\rightarrow$ `for` 문 종료!
  4. **★ [수험생과의 질답 Q&A 필기 노트] 주요 언어 3종 (C vs Java vs Python) 논리 부정 !(!a) 반환 타입 비교 족보**:
     * **C 언어**: 논리 연산의 결과가 정수(int) 타입 **`1`** 또는 **`0`** 으로 변환되어 저장됨! (원본 숫자 3으로 돌아가지 않고 `1` 이 됨)
     * **Java 언어**: 정수형 변수 `int a = 3;` 에 논리 연산자 `!a` 를 취하면 **컴파일 에러 (Compile Error)** 발생! (`boolean` 변수 `boolean a = true;` 에 `!(!a)` 를 취해야 **`true`** 로 원상 복귀됨)
     * **Python 언어**: 숫자 `a = 3` 에 `not(not a)` 를 취하면 원본 숫자 3이 아니라 불리언 타입인 **`True`** 값으로 환산됨!

## 156. 서비스 운영 및 인프라 구축 - 응용 소프트웨어 기초 기술 (참조의 국소성 Locality 3대 유형 공간 Spatial 시간 Temporal 순차 Sequential 국부성)

### 문제
다음 국부성(Locality) 유형 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① ) locality는 프로세스 실행 시 일정 위치의 페이지를 집중적으로 액세스하는 현상으로 사례로는 배열 순회 등이 있다.
* ( ② ) locality는 최근 사용되었던 기억장소들이 집중적으로 액세스하는 현상으로 사례로는 Loop, Stack, Sub Routine 등이 있다.
* ( ③ ) locality는 데이터가 순차적으로 액세스 되는 현상으로 사례로는 순차적 코드 실행이 있다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 공간 (Spatial)**
  * **②: 시간 (Temporal)**
  * **③: 순차 (Sequential)**
* **모범 답안**:
  * **①: 공간** (또는 `공간 국부성`, `Spatial`)
  * **②: 시간** (또는 `시간 국부성`, `Temporal`)
  * **③: 순차** (또는 `순차 국부성`, `Sequential`)
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 컴퓨터 시스템의 주기억장치 및 캐시 메모리(Cache Memory)의 효율적 운용과 가상 메모리 관리의 기초가 되는 **참조의 국소성(Locality of Reference) 3대 개념인 공간 국부성, 시간 국부성, 순차 국부성의 대표적 사례와 특징**을 상호 구분할 수 있는지 검증하는 단답형 기출 문제입니다.

* **국부성(Locality) 3대 유형 정밀 분석 및 대표 사례 족보**:
  1. **공간 국부성 (Spatial Locality) (지문 ①번)**:
     * *정의*: 프로세스 실행 시 기억장치의 특정 위치 근처(주변 장소)에 인접한 데이터들이 연속적으로 액세스되는 특성.
     * *대표 사례*: **배열 순회(Array Traversal)**, 인접 변수 선언/참조, 테이블 요소 접근.
  2. **시간 국부성 (Temporal Locality) (지문 ②번)**:
     * *정의*: 최근에 한번 참조된 기억장소의 데이터가 가까운 미래에 또다시 집중적으로 재참조되는 특성.
     * *대표 사례*: **반복문 루프(Loop: for/while)**, **스택(Stack)**, **부프로그램(Subroutine)**, 카운팅(Counting) 변수.
  3. **순차 국부성 (Sequential Locality) (지문 ③번)**:
     * *정의*: 프로그램의 데이터나 명령어가 메모리에 저정된 순서대로 차례차례 순차적으로 실행되는 특성.
     * *대표 사례*: **순차적 코드 실행(Sequential Code Execution)**, 명령 프로그램 카운터(PC)의 순차 증가.

## 157. 서비스 운영 및 인프라 구축 - 소프트웨어 개발 보안 구축 (재해 복구 시스템 DRS 대기 센터 4대 유형 Hot Site vs Warm Site 복구 시간 RTO 기준 구별)

### 문제
다음 재해 복구 센터(DRS) 유형 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① )은/는 주 센터와 동일한 수준의 자원을 대기 상태로 원격지에 보유하면서 동기, 비동기 방식의 미러링을 통하여 데이터의 최신 상태를 유지하고 있는 재해복구센터로 RTO는 4시간 이내이다.
* ( ② )은/는 ( ① )와/과 유사하나 재해복구센터에 주 센터와 동일한 수준의 자원을 보유하는 대신 중요성이 높은 자원만 부분적으로 재해복구센터에 보유하고 있는 센터로 RTO는 수일~수주이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: Hot Site**
  * **②: Cold Site** (부분적 중요 자원 보유 및 수일~수주 RTO 스펙을 최하위 인프라 부지 확보 단계인 Cold Site로 착각하여 ②번 오답 처리)
* **모범 답안**:
  * **①: Hot Site (핫 사이트)**
  * **②: Warm Site (웜 사이트)**
* **결과**: **부분 정답 (①번 정답, ②번 웜 사이트 Warm Site 오답)**

### 핵심 해설
이 문제는 정보시스템 재해 복구 관리(Disaster Recovery System) 영역에서 비즈니스 연속성(BCP) 확보를 위해 구축하는 **DRS 4대 대기 복구 센터(Mirror Site, Hot Site, Warm Site, Cold Site)의 자원 보유 수준 및 목표 복구 시간(RTO: Recovery Time Objective)**을 구별할 수 있는지 평가하는 기출 단답형 문제입니다.

* **DRS 4대 대기 복구 센터 정밀 스펙 비교 족보**:
  1. **`Mirror Site (미러 사이트)`**:
     * *자원 수준*: 주 센터와 동일한 시스템 및 데이터를 실시간 복제(Active-Active).
     * *RTO*: **`0 (즉시 복구)`** / 데이터 손실 0.
  2. **`Hot Site (핫 사이트) (지문 ①번)`**:
     * *자원 수준*: 주 센터와 **동일한 수준의 자원을 대기 상태(Active-Standby)**로 보유하며 동기/비동기 실시간 미러링 수행.
     * *RTO*: **`수시간 이내 (보통 4시간 이내)`**.
  3. **`Warm Site (웜 사이트) (지문 ②번)`**:
     * *자원 수준*: 주 센터 자원 전체 대신 **중요성이 높은 자원만 '부분적으로' 보유**하고 백업 데이터를 수송/복구함.
     * *RTO*: **`수일 ~ 수주 이내`**.
  4. **`Cold Site (콜드 사이트)`**:
     * *자원 수준*: 데이터 복구 시설의 **기본 부지(장소), 전력/통신 기반 시설만 확보**해 두고 재해 시 신규 장비를 새로 구매 및 배치함.
     * *RTO*: **`수주 ~ 수개월 이내`**.

## 158. 서비스 운영 및 인프라 구축 - 소프트웨어 개발 보안 구축 (경유지 반사 서버 Reflector 및 IP 스푸핑 기반 DRDoS 분산 반사 서비스 거부 공격)

### 문제
다음 3단계 보안 공격 기법 설명에 해당하는 보안 공격 명칭을 쓰시오.
* **1단계**: 공격자는 많은 수의 경유지 서버에 특정 질의 명령어(요청)를 일괄적으로 보냄. 자신의 IP를 공격 대상 IP로 변조(IP Spoofing)하여 공격 대상 IP가 응답받도록 하고, 효과적인 공격을 위하여 경유지 서버가 응답하는 양이 큰 질의 명령어를 사용함.
* **2단계**: 경유지 서버는 정상 요청으로 인식하여 공격 대상 서버에 명령어 결과(응답)를 보냄.
* **3단계**: 많은 수의 경유지 서버로부터 응답을 받은 공격 대상 네트워크는 트래픽이 급증하여 회선 대역폭이 가득 차게 됨.

### 정답 및 풀이 결과
* **작성한 답**: `DDoS (분산 서비스 거부 공격)` (분산된 트래픽 공격의 대분류인 DDoS 개념에 90% 이상 정밀 접근했으나, 경유지 반사체 Reflector 및 IP 스푸핑 증폭 메커니즘을 구별하는 파생 기술명 DRDoS 기재 누락으로 오답 처리)
* **모범 답안**: **`DRDoS`** (또는 `분산 반사 서비스 거부 공격`, `DRDoS (분산 반사 서비스 거부 공격)`)
* **결과**: **오답 (일반 DDoS와 반사체 Reflector 기반 DRDoS 세부 구별 실패)**

### 핵심 해설
이 문제는 정보보안 및 소프트웨어 개발 보안 영역의 대표적인 서비스 거부 공격(DoS/DDoS) 파생 유형 중 **경유지 서버(Reflector)를 반사체로 이용하고 출발지 IP를 변조(IP Spoofing)하여 응답 트래픽을 증폭시켜 공격하는 DRDoS(Distributed Reflective Denial of Service, 분산 반사 서비스 거부 공격)** 의 동작 메커니즘을 식별할 수 있는지 평가하는 기출 단답형 문제입니다.

* **DDoS vs DRDoS 2대 킬러 차이점 족보**:
  1. **`일반 DDoS (Distributed Denial of Service)`**:
     * *방식*: 공격자가 악성코드에 감염된 **좀비 PC(Botnet)** 들을 직접 조종하여 공격 대상 서버로 **직접(Direct)** 대량의 트래픽을 쏘아 보냄.
  2. **`DRDoS (Distributed Reflective Denial of Service) (본 문제)`**:
     * *방식*: 공격자가 자신의 IP를 공격 대상의 IP로 속이는 **`IP 스푸핑(IP Spoofing)`** 을 적용한 후, 정상 작동하는 무수히 많은 **`경유지 서버(Reflector, 반사체)`** 로 소량의 질의 요청을 보냄.
     * *결과*: 경유지 서버들이 반사적으로 **엄청난 크기의 응답(증폭 트래픽)** 을 공격 대상 서버로 한꺼번에 쏟아부어 공격 대상의 네트워크 대역폭을 마비시킴.

## 159. 서비스 운영 및 인프라 구축 - 응용 소프트웨어 기초 기술 (국가망 및 광역 공중통신망 상호 접속 커버리지 WAN Wide Area Network 광역 통신망)

### 문제
다음 설명에 해당하는 네트워크 종류(용어)를 쓰시오.
* ( ) 은/는 국가망 또는 각 국가의 공중통신망을 상호 접속시키는 국제정보통신망으로 설계 및 구축, 운용되는 네트워크이다.
* 공중통신망 사업자가 구축하고, 일반 대중 가입자들에게 보편적인 정보통신 서비스를 제공한다.
* 거리에 제약이 없으나 다양한 경로를 지나 정보가 전달되므로 LAN보다 속도가 느리고 에러율도 높다.

### 정답 및 풀이 결과
* **작성한 답**: `VPN` (공중망을 전용선처럼 암호화 터널링하여 사설망으로 활용하는 보안 네트워크인 VPN 개념으로 잘못 오인하여 오답 처리)
* **모범 답안**: **`WAN`** (또는 `광역 통신망`, `Wide Area Network`, `WAN (광역 통신망)`)
* **결과**: **오답 (네트워크 커버리지 개념인 WAN과 보안 터널링 기술인 VPN 상호 혼동)**

### 핵심 해설
이 문제는 컴퓨터 네트워크 관리 및 인프라 영역에서 **물리적 커버리지(전송 거리 및 광역 범위)에 따른 네트워크 분류인 WAN(Wide Area Network, 광역 통신망)** 의 고유 특성과 명세를 구별할 수 있는지 검증하는 빈출 단답형 문제입니다.

* **네트워크 물리 커버리지 4대 분류 족보**:
  1. **`PAN (Personal Area Network, 개인 영역 네트워크)`**:
     * *범위*: 10m 이내 개인 최단 거리. (Bluetooth, UWB, Zigbee)
  2. **`LAN (Local Area Network, 근거리 통신망)`**:
     * *범위*: 사무실, 학교, 집 등 수 km 이내의 한정된 지역. 높은 전송 속도 및 낮은 에러율. (Ethernet, Wi-Fi)
  3. **`MAN (Metropolitan Area Network, 도시권 통신망)`**:
     * *범위*: 대도시 영역(수십 km 이내). LAN과 WAN의 중간 연결체. (FDDI, DQDB)
  4. **`WAN (Wide Area Network, 광역 통신망) (본 문제)`**:
     * *범위*: **국가, 대륙, 전 세계를 상호 접속시키는 광범위 통신망.**
     * *특징*: 공중통신망 사업자(KT, SKT 등)가 구축하며, **거리 제약이 없으나 LAN에 비해 속도가 느리고 에러율이 상대적으로 높음.**

* **오답 파해: VPN (Virtual Private Network, 가상 사설망) 과의 차이**:
  * **VPN**은 커버리지 종류가 아니라, 인터넷과 같은 공중망(Public Network)을 마치 암호화된 전용선(Private Line)처럼 안전하게 사용할 수 있도록 **터널링(Tunneling) 및 암호화(Encryption)**를 적용하는 **보안 기술**입니다.

## 160. 서비스 운영 및 인프라 구축 - 소프트웨어 개발 보안 구축 (소프트웨어 개발사 배포 업데이트 서버 침투 및 파일 변조 기반 공급망 공격 Supply Chain Attack)

### 문제
다음 설명에 해당하는 보안 공격 기법 명칭을 쓰시오.
* 소프트웨어 개발사의 네트워크에 침투하여 소스 코드의 수정 등을 통해 악의적인 코드를 삽입하거나 배포 서버에 접근하여 악의적인 파일로 변경하는 방식을 통해 사용자 PC에 소프트웨어를 설치 또는 업데이트 시에 자동적으로 감염되도록 하는 공격 기법

### 정답 및 풀이 결과
* **작성한 답**: `웜 (Worm)` (네트워크를 통해 스스로 복제 전파되는 악성 바이러스 프로그램인 웜 개념으로 잘못 오인하여 오답 처리)
* **모범 답안**: **`공급망 공격`** (또는 `Supply Chain Attack`, `공급망 공격 (Supply Chain Attack)`)
* **결과**: **오답 (소프트웨어 개발사/업데이트 서버 경로 오염 기반 공급망 공격과 악성코드 웜 상호 구분 실패)**

### 핵심 해설
이 문제는 소프트웨어 개발 보안 및 인프라 영역에서 최근 핫이슈로 빈출되는 **신뢰된 소프트웨어 개발사나 배포/업데이트 서버(Supply Chain)를 먼저 오염시켜 해당 소프트웨어를 다운로드받는 대규모 정식 사용자 PC를 연쇄 감염시키는 공급망 공격(Supply Chain Attack)** 의 동작 메커니즘을 구별할 수 있는지 평가하는 단답형 기출 문제입니다.

* **공급망 공격(Supply Chain Attack)의 핵심 메커니즘 정밀 분석**:
  1. **침투 타겟**: 사용자 PC가 아닌, 사용자들이 100% 신뢰하는 **`소프트웨어 개발사`**, **`업데이트 서버`**, **`하드웨어 제조업체`** 를 1차 타겟으로 침투합니다.
  2. **오염 방식**: 정식 업데이트 파일이나 소스 코드에 악성코드를 숨겨 놓습니다(백도어/악성 모듈 변조).
  3. **파급력**: 사용자가 정식 업데이트 버튼을 누르는 순간 정상 파일로 위장하여 자동 감염되므로 기존 백신(Vaccine) 필터링을 완벽히 무력화시키는 위협을 가집니다.
  4. *대표 사례*: SolarWinds 사태, 3CX 전화 소프트웨어 업데이트 감염 사태.

* **오답 파해: 웜 (Worm) 과의 확실한 구분점**:
  * **`웜 (Worm)`**: 다른 정상 파일이나 서버에 의존하지 않고, 독자적으로 작동하여 네트워크를 통해 **스스로를 자가 복제(Self-Replication)**하면서 확산되는 악성 프로그램입니다.
  * 지문처럼 **"소프트웨어 개발사의 배포/업데이트 서버 경로를 오염시켜 감염"**시키는 공격 형태는 악성코드 종류가 아닌 **`공급망 공격 (Supply Chain Attack)`** 이 정답입니다.

## 161. Python - 프로그래밍 언어 활용 (문자열 슬라이싱 word[::-1] 기반 팰린드롬 Palindrome 회문 단어 판별 및 최장 길이 len 갱신 출력)

### 문제
아래 파이썬 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 코드 블록 HTML 내에서 바르게 파이썬 코드로 디해싱하여 풀이한다.)
```python
def soojebi(text):
  str = ""
  for word in text.split():
    if word == word[::-1]:
      if len(word) > len(str):
        str = word
  if str:
    print(f"{str}")
  else:
    print("No found")

text = "banana racecar apple madam noon civic rotor"
soojebi(text)
```

### 정답 및 풀이 결과
* **작성한 답**: `madam` (슬라이싱 word[::-1] 이 팰린드롬(회문)을 판별한다는 원리는 파악했으나, 문자열 앞부분의 최장 단어인 racecar(길이 7)가 madam(길이 5)보다 길어 len 갱신 조건을 충족하지 못함을 놓쳐 오답 처리)
* **모범 답안**: **`racecar`**
* **결과**: **오답 (팰린드롬 판별 성공 후 len(word) > len(str) 최장 단어 길이 비교 갱신 누락)**

### 핵심 해설
이 문제는 파이썬의 핵심 문자열 조작 문법인 **슬라이싱 스텝 `word[::-1]` 을 활용한 팰린드롬(Palindrome, 회문) 검사 로직**과, **`len(word) > len(str)` 조건식을 통한 최장 단어 길이 선택 알고리즘**을 정확하게 트레이싱할 수 있는지 평가하는 문제입니다.

* **단계별 단어 트레이싱 분석**:
  1. `text.split()` ➔ `['banana', 'racecar', 'apple', 'madam', 'noon', 'civic', 'rotor']` 리스트로 분할됩니다.
  2. **`word = 'banana'`**: `'banana' == 'ananab'` $\rightarrow$ **False (거짓)**
  3. **`word = 'racecar'`**:
     * `'racecar' == 'racecar'` $\rightarrow$ **True (참: 팰린드롬 단어!)**
     * `len('racecar')` (7) $>$ `len(str)` (0) $\rightarrow$ **True**
     * **`str = 'racecar'`** 로 갱신됩니다! (현재 최장 길이 = 7)
  4. **`word = 'apple'`**: `'apple' == 'elppa'` $\rightarrow$ **False (거짓)**
  5. **`word = 'madam'`**:
     * `'madam' == 'madam'` $\rightarrow$ **True (참: 팰린드롬 단어!)**
     * `len('madam')` (5) $>$ `len(str)` (7) $\rightarrow$ **False (거짓!)**
     * 길이 5는 기존 최장인 7보다 작으므로 `str` 변수가 갱신되지 않고 그대로 **`'racecar'`** 가 유지됩니다!
  6. **`word = 'noon'`, `'civic'`, `'rotor'`**: 모두 팰린드롬이지만 길이가 각각 4, 5, 5 로 기존 7보다 작아 갱신되지 않습니다.
  7. **최종 출력**: `str` 인 **`racecar`** 가 화면에 인쇄됩니다.

## 162. 소프트웨어 개발 - 소프트웨어 공학 (객체지향 설계 5대 원칙 SOLID 중 단일 책임 원칙 SRP 및 개방 폐쇄 원칙 OCP)

### 문제
다음 객체지향 설계 원칙 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① )은/는 하나의 클래스는 하나의 목적을 위해서 생성되며, 클래스가 제공하는 모든 서비스는 하나의 책임을 수행하는 데 집중되어 있어야 한다는 원칙이다.
* ( ② )은/는 소프트웨어의 구성요소(컴포넌트, 클래스, 모듈, 함수)는 확장에는 열려있고, 변경에는 닫혀있어야 한다는 원칙이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 싱글톤** (SOLID 원칙의 S인 단일 책임 원칙 SRP를 GoF 디자인 패턴인 싱글톤 Singleton으로 착각하여 오답)
  * **②: 열림 닫힘 원칙** (OCP 개방 폐쇄 원칙의 개념적 핵심을 바르게 떠올려 100% 정답 인정)
* **모범 답안**:
  * **①: 단일 책임 원칙** (또는 `SRP`, `Single Responsibility Principle`)
  * **②: 개방 폐쇄 원칙** (또는 `OCP`, `Open-Closed Principle`, `개방-폐쇄 원칙`)
* **결과**: **부분 정답 (①번 오답, ②번 개방 폐쇄 원칙 정답)**

### 핵심 해설
이 문제는 객체지향 설계 및 소프트웨어 공학 영역의 최빈출 뼈대 이론인 **객체지향 5대 설계 원칙 (SOLID 원칙)** 중 **단일 책임 원칙(SRP)** 과 **개방 폐쇄 원칙(OCP)** 의 정의와 명세를 식별할 수 있는지 평가하는 기출 단답형 문제입니다.

* **객체지향 5대 설계 원칙 (SOLID) 완전 해체 족보**:
  1. **`S: 단일 책임 원칙 (SRP - Single Responsibility Principle) (지문 ①번)`**:
     * *개념*: 하나의 클래스는 **오직 단 하나의 책임(하나의 목적)**만을 가져야 함. 클래스를 변경해야 하는 이유는 오직 하나뿐이어야 함.
  2. **`O: 개방 폐쇄 원칙 (OCP - Open-Closed Principle) (지문 ②번)`**:
     * *개념*: **확장에는 열려(Open) 있고, 변경(수정)에는 닫혀(Closed) 있어야 한다**는 원칙. 기존 코드를 수정하지 않고 새로운 기능을 추가할 수 있도록 인터페이스를 추상화함.
  3. **`L: 리스코프 치환 원칙 (LSP - Liskov Substitution Principle)`**:
     * *개념*: 자식 클래스(하위 타입)는 언제나 부모 클래스(상위 타입)를 대체할 수 있어야 함. (부모 타입 객체 자리에 자식 객체를 넣어 놓아도 정상 가동됨)
  4. **`I: 인터페이스 분리 원칙 (ISP - Interface Segregation Principle)`**:
     * *개념*: 하나의 범용 인터페이스보다 클라이언트가 실제로 사용하는 여러 개의 구체적인 인터페이스로 분리해야 함. (자신이 사용하지 않는 메서드에 의존하지 않음)
  5. **`D: 의존 역전 원칙 (DIP - Dependency Inversion Principle)`**:
     * *개념*: 구체적인 구현 클래스에 의존하지 말고, 변하지 않는 **추상화(인터페이스)**에 의존해야 함.

* **오답 파해: 싱글톤 (Singleton) 과의 명확한 차이**:
  * **`싱글톤 (Singleton)`**: 객체지향 5대 설계 원칙(SOLID)이 아니라, GoF 디자인 패턴(Design Pattern) 중 생성 패턴의 하나로서 **"전체 시스템에서 인스턴스(객체)를 오직 단 1개만 생성하여 공유"**하는 클래스 생성 패턴입니다.

## 163. 데이터베이스 구축 - SQL 응용 및 데이터베이스 기초 (식별자 및 키 Key 2대 핵심 제약조건 유일성 Uniqueness 및 최소성 Minimality)

### 문제
다음 식별자(Key)의 제약 특성 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① )은/는 식별자에 의해 엔터티 내에 모든 튜플들을 유일하게 구분하는 특성이다.
* ( ② )은/는 최소한의 속성으로 식별자를 구성하는 특성이다.

### 정답 및 풀이 Result
* **작성한 답**:
  * **①: 원자성** (트랜잭션 ACID 4대 특성 중 하나인 원자성 Atomicity 개념으로 잘못 오인하여 오답)
  * **②: 최소성** (식별자를 구성하는 최소한의 속성 특성을 정밀 파악하여 100% 정답)
* **모범 답안**:
  * **①: 유일성** (또는 `Uniqueness`, `유일성 (Uniqueness)`)
  * **②: 최소성** (또는 `Minimality`, `최소성 (Minimality)`)
* **결과**: **부분 정답 (①번 오답, ②번 최소성 정답)**

### 핵심 해설
이 문제는 데이터베이스(RDB) 설계 및 모델링 영역에서 후보키, 기본키, 슈퍼키를 정의하는 핵심 제약조건인 **유일성(Uniqueness)** 과 **최소성(Minimality)** 의 정의를 구별할 수 있는지 평가하는 기출 단답형 문제입니다.

* **DB 키(Key)의 2대 제약 특성 정밀 비교 족보**:
  1. **`유일성 (Uniqueness) (지문 ①번)`**:
     * *개념*: 하나의 키 값으로 릴레이션(테이블) 내에 존재하는 **모든 튜플(행)을 서로 구별하여 오직 단 하나만 유일하게 식별할 수 있는 성질.**
     * *만족하는 키*: 슈퍼키(Super Key), 후보키(Candidate Key), 기본키(Primary Key), 대체키(Alternate Key).
  2. **`최소성 (Minimality) (지문 ②번)`**:
     * *개념*: 튜플을 유일하게 구별하기 위해 필요한 속성(컬럼)들의 집합 중 **더 이상 제거할 수 없는 꼭 필요한 최소한의 속성으로만 구성되는 성질.**
     * *만족하는 키*: 후보키(Candidate Key), 기본키(Primary Key), 대체키(Alternate Key). (※ 슈퍼키는 유일성은 만족하지만 최소성은 만족하지 못함!)

* **오답 파해: 원자성 (Atomicity) 과의 명확한 차이**:
  * **`원자성 (Atomicity)`**: 데이터베이스 **트랜잭션(Transaction)의 4대 특성(ACID)** 중 하나로서 "트랜잭션의 연산은 DB에 모두 반영되거나(All) 전혀 반영되지 않아야 함(Nothing)"을 뜻하는 성질입니다. 튜플 구별 제약조건인 **`유일성`** 과 완전히 다른 영역의 단어입니다.

* **★ [수험생 질답 심화 필기 노트] 식별자(Entity Identifier)의 4대 특성 완전 확장 족보**:
  1. **`유일성 (Uniqueness)`**: 엔티티 내의 모든 튜플을 유일하게 구별함. (본 문제 ①번)
  2. **`최소성 (Minimality)`**: 튜플을 유일하게 구별하는 최소한의 속성으로 구성됨. (본 문제 ②번)
  3. **`불변성 (Immutability)`**: 한번 식별자로 지정되면 식별자의 값은 변하지 않아야 함. (예: 학번, 주민번호 등)
  4. **`존재성 (Existence / Non-null)`**: 주 식별자는 반드시 데이터 값이 존재해야 함. (Null 값 절대 불가)

## 164. 소프트웨어 개발 - 요구사항 확인 및 객체지향 분석 (UML 클래스 다이어그램 간 관계 중 합성 Composition 및 집합 Aggregation)

### 문제
다음 UML 클래스 다이어그램 간 관계 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① ) 관계는 포함하는 사물의 변화가 포함되는 사물에게 영향을 미치는 관계를 표현하고 복합 관계라고도 한다.
* ( ① ) 관계는 포함되는 쪽(부분)에서 포함하는 쪽(전체)으로 속이 채워진 마름모를 연결하여 표현한다.
* ( ② ) 관계는 하나의 객체에 여러 개의 독립적인 객체들이 구성되는 관계로 하나의 사물이 다른 사물에 포함되어 있는 관계를 표현한다.
* ( ② ) 관계는 포함되는 쪽(부분)에서 포함하는 쪽(Whole)으로 속이 빈 마름모를 연결하여 표현한다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 포함** (Composition에 해당하는 번역 용어로 '포함', '합성', '복합' 모두 정답 인정)
  * **②: 일반화** (속이 빈 마름모 도형 표기를 is-a 상속 관계인 일반화 Generalization 속이 빈 화살표와 착각하여 오답)
* **모범 답안**:
  * **①: 합성** (또는 `복합`, `포함`, `Composition`, `합성 관계`)
  * **②: 집합** (또는 `Aggregation`, `집합 관계`)
* **결과**: **부분 정답 (①번 포함 정답 인정, ②번 일반화 오답)**

### 핵심 해설
이 문제는 소프트웨어 요구사항 분석 및 설계 영역에서 UML(Unified Modeling Language) 클래스 다이어그램의 **클래스 간 관계(Relationships) 중 전체(Whole)와 부분(Part)의 포함 관여성을 나타내는 합성 관계(Composition)와 집합 관계(Aggregation)의 개념 및 그래픽 표기 도형(마름모)**을 구별할 수 있는지 검증하는 기출 문제입니다.

* **합성(Composition) vs 집합(Aggregation) 킬러 비교 족보**:
  1. **`합성 / 복합 관계 (Composition) (지문 ①번)`**:
     * *개념*: 전체(Whole)와 부분(Part)의 **생명주기(Lifecycle)가 완전히 일치**하는 관계. 전체 객체가 소멸되면 부분 객체도 함께 소멸됨. (강한 결합)
     * *표기 도형*: 부분에서 전체 쪽으로 **`속이 채워진 검은 마름모 (◆)`** 로 연결함.
  2. **`집합 관계 (Aggregation) (지문 ②번)`**:
     * *개념*: 전체(Whole)와 부분(Part)이 **독립적인 생명주기**를 가짐. 전체 객체가 소멸되어도 부분 객체는 독립적으로 존재함. (약한 결합)
     * *표기 도형*: 부분에서 전체 쪽으로 **`속이 빈 흰색 마름모 (◇)`** 로 연결함.

* **UML 클래스 간 6대 관계 도형 표기법 완전 정리**:
  * **연관 (Association)**: 실선 (`──`) 또는 실선 화살표 (`──>`)
  * **의존 (Dependency)**: 점선 화살표 (`- - - >`)
  * **집합 (Aggregation)**: **속이 빈 마름모 (`◇──`)** (지문 ②번)
  * **합성 (Composition)**: **속이 채워진 마름모 (`◆──`)** (지문 ①번)
  * **일반화 (Generalization)**: **속이 빈 삼각 화살표 (`──▷`)** (상속 is-a 관계)
  * **실체화 (Realization)**: **점선 속이 빈 삼각 화살표 (`- - -▷`)** (인터페이스 구현)

## 165. 소프트웨어 개발 - 요구사항 확인 및 아키텍처 설계 (소프트웨어 아키텍처 패턴 중 파이프-필터 패턴 Pipe-Filter Pattern 및 브로커 패턴 Broker Pattern)

### 문제
다음 소프트웨어 아키텍처 패턴 설명의 괄호 ( ) 안에 들어갈 알맞은 용어를 쓰시오.
* ( ① )은/는 데이터 스트림을 생성하고 처리하는 시스템에서 사용 가능한 단방향 패턴으로 서브 시스템이 입력 데이터를 받아 처리하고, 결과를 다음 서브 시스템으로 넘겨주는 과정을 반복하는 방식이다.
* ( ② )은/는 분리된 컴포넌트들로 이루어진 분산 시스템에서 사용되고, 이 컴포넌트들은 원격 서비스 실행을 통해 상호 작용이 가능한 패턴으로 컴포넌트 간의 통신을 조정하는 역할을 수행하는 방식이다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 파이프 필터 패턴** (데이터 스트림 처리 아키텍처 패턴의 표준 명칭을 100% 바르게 도출해 정답)
  * **②: 제어 패턴** (분산 컴포넌트 간의 원격 통신 조정 및 중재 매개체인 브로커 패턴 Broker Pattern 명칭을 잘못 오인하여 오답)
* **모범 답안**:
  * **①: 파이프-필터 패턴** (또는 `파이프 필터`, `Pipe-Filter Pattern`, `파이프-필터`)
  * **②: 브로커 패턴** (또는 `Broker Pattern`, `브로커`)
* **결과**: **부분 정답 (①번 파이프 필터 패턴 정답, ②번 제어 패턴 오답)**

### 핵심 해설
이 문제는 소프트웨어 구조 설계(Architecture Design) 영역에서 시스템 전체의 뼈대를 형성하는 **주요 소프트웨어 아키텍처 패턴(Software Architecture Patterns) 4대 요소인 파이프-필터 패턴과 브로커 패턴의 핵심 명세 및 역할을 구별**할 수 있는지 검증하는 빈출 기출 문제입니다.

* **소프트웨어 아키텍처 패턴 4대 대표 유형 완전 해체 족보**:
  1. **`파이프-필터 패턴 (Pipe-Filter Pattern) (지문 ①번)`**:
     * *개념*: **데이터 스트림(Data Stream)** 을 처리하는 서브시스템 구축에 활용. 데이터 가공/변환을 담당하는 **필터(Filter)** 들을 데이터 전송 통로인 **파이프(Pipe)** 로 단방향 연결함.
     * *대표 사례*: UNIX / Linux CLI 쉘의 파이프 명령어(`cat file.txt | grep "error" | sort`).
  2. **`브로커 패턴 (Broker Pattern) (지문 ②번)`**:
     * *개념*: **분산 시스템(Distributed System)** 환경에서 클라이언트와 서버 사이의 통신 및 원격 서비스 실행을 **중재(Broker)** 해 주는 패턴. 서버는 자신의 서비스를 브로커에게 등록하고, 클라이언트는 브로커를 통해 서비스를 요청함.
     * *대표 사례*: CORBA, Java RMI.
  3. **`MVC 패턴 (Model-View-Controller Pattern)`**:
     * *개념*: 시스템을 비즈니스 로직(**Model**), 사용자 화면 UI(**View**), 흐름 제어/중재자(**Controller**)의 3가지 서브시스템으로 분리함.
  4. **`피어 투 피어 패턴 (Peer-to-Peer Pattern)`**:
     * *개념*: 각 컴포넌트(Peer)가 클라이언트이면서 동시에 서버 역할을 모두 수행하는 대등한 분산 아키텍처 패턴.

## 166. 서비스 운영 및 인프라 구축 - 응용 소프트웨어 기초 기술 (OSI 7계층 중 2계층 데이터 링크 계층 Data Link Layer 프로토콜 ㉢ PPP 및 ㉦ HDLC 식별)

### 문제
다음 보기 중에서 **데이터 링크 계층(Data Link Layer)**에 해당하는 프로토콜만을 모두 고르시오.
```text
㉠ UDP                 ㉡ IP                   ㉢ PPP 
㉣ HTTP               ㉤ ARP               ㉥ FTP
㉦ HDLC              ㉧ TCP
```

### 정답 및 풀이 결과
* **작성한 답**: `㉠, ㉤, ㉦` (4계층 전송 계층인 UDP와 3계층 네트워크 계층인 ARP를 2계층으로 잘못 포함시키고, 직렬 점대점 2계층 프로토콜인 ㉢ PPP를 선택에서 누락하여 오답)
* **모범 답안**: **`㉢, ㉦`** (또는 `㉢ PPP`, `㉦ HDLC`, `ㄷ, ㅅ`)
* **결과**: **오답 (계층별 프로토콜 식별 미흡: ㉢ PPP 누락, ㉠ UDP 및 ㉤ ARP 착각)**

### 핵심 해설
이 문제는 네트워크 아키텍처(OSI 7계층 및 TCP/IP 4계층) 영역에서 각 계층별로 작동하는 **대표적인 통신 프로토콜(Protocols)의 계층적 위치**를 정확하게 식별할 수 있는지 검증하는 기출 단답형 보기 선택 문제입니다.

* **보기 8개 프로토콜의 OSI 7계층 위치 완전 해체 족보**:
  1. **`㉢ PPP (Point-to-Point Protocol)`**: **`2계층 (데이터 링크 계층)`** ➔ **[정답]**
     * *역할*: 1:1 직렬 통신 회선에서 데이터 전송, 에러 검출, 에선 프레임을 형성하는 표준 데이터 링크 프로토콜.
  2. **`㉦ HDLC (High-level Data Link Control)`**: **`2계층 (데이터 링크 계층)`** ➔ **[정답]**
     * *역할*: 비트(Bit) 위주의 고속 표준 데이터 링크 제어 프로토콜. (명칭 자체에 Data Link 가 들어있음!)
  3. **`㉠ UDP (User Datagram Protocol)`**: `4계층 (전송 계층)`
     * *역할*: 비연결형, 신뢰성은 없으나 빠른 전송을 담당하는 4계층 프로토콜.
  4. **`㉡ IP (Internet Protocol)`**: `3계층 (네트워크 계층)`
     * *역할*: 패킷 주소 지정 및 라우팅(경로 설정)을 담당하는 3계층 메인 프로토콜.
  5. **`㉣ HTTP (HyperText Transfer Protocol)`**: `7계층 (응용 계층)`
     * *역할*: 웹 브라우저와 서버 간 문서 전송 7계층 프로토콜.
  6. **`㉤ ARP (Address Resolution Protocol)`**: `3계층 (네트워크 계층)`
     * *역할*: 논리적 IP 주소를 물리적 MAC 주소로 변환하는 3계층 연동 프로토콜.
  7. **`㉥ FTP (File Transfer Protocol)`**: `7계층 (응용 계층)`
     * *역할*: 파일 전송 담당 7계층 프로토콜.
  8. **`㉧ TCP (Transmission Control Protocol)`**: `4계층 (전송 계층)`
     * *역할*: 신뢰성 있는 연결형 전송 담당 4계층 프로토콜.

## 167. 서비스 운영 및 인프라 구축 - 응용 소프트웨어 기초 기술 (IPv4 vs IPv6 4대 규격 스펙 주소 비트 수 및 헤더 바이트 크기 비교)

### 문제
다음 IPv4 및 IPv6 프로토콜 설명의 괄호 ( ) 안에 들어갈 알맞은 숫자를 쓰시오.
* IPv4는 인터넷에서 사용되는 패킷 교환 네트워크상에서 데이터를 교환하기 위한 ( ① )비트 주소 체계를 갖는 네트워크 계층의 프로토콜이다.
* IPv4 헤더(Header)는 IP 패킷의 앞부분에서 주소 등 각종 제어정보를 담고 있으며 IPv4 헤더 사이즈는 옵션 미 지정시에 최소 ( ② )바이트 이상이다.
* IPv6는 ( ③ )비트의 주소 공간을 제공한다.
* IPv6 주소의 경우 일반적으로 16비트 단위로 나누어지며 각 16비트 블록은 다시 4자리 16진수로 변환되고 콜론으로 구분된다.
* IPv6 헤더 사이즈는 ( ④ )바이트의 고정된 길이로 되어 있다.

### 정답 및 풀이 결과
* **작성한 답**:
  * **①: 128** (IPv4 32비트와 IPv6 128비트 위치 착각 오답)
  * **②: 4** (IPv4 최소 헤더 20바이트 오답)
  * **③: 256** (IPv6 128비트 오답)
  * **④: 6** (IPv6 고정 헤더 40바이트 오답)
* **모범 답안**:
  * **①: 32**
  * **②: 20**
  * **③: 128**
  * **④: 40**
* **결과**: **오답 (IPv4/IPv6 주소 비트 수 및 헤더 바이트 숫자 스펙 혼동)**

### 핵심 해설
이 문제는 컴퓨터 네트워크 영역에서 인터넷의 기본 주소 체계인 **IPv4 와 차세대 주소 체계인 IPv6 의 4대 핵심 숫자 규격 스펙(주소 비트 길이, 헤더 바이트 크기)**을 정확히 구분하고 암기하고 있는지 검증하는 빈출 기출 수치 계산 문제입니다.

* **IPv4 vs IPv6 4대 핵심 수치 스펙 완전 해체 족보**:
  1. **IPv4 주소 비트 수 (지문 ①번)**: **`32 비트`** (8비트씩 4부분 = 32bits, 10진수로 표기하며 마침표 `.` 로 구분)
  2. **IPv4 최소 헤더 크기 (지문 ②번)**: **`20 바이트 (Bytes)`** (옵션 미지정 시 최소 20 Bytes ~ 최대 60 Bytes 가변적)
  3. **IPv6 주소 비트 수 (지문 ③번)**: **`128 비트`** (16비트씩 8부분 = 128bits, 16진수로 표기하며 콜론 `:` 로 구분)
  4. **IPv6 고정 헤더 크기 (지문 ④번)**: **`40 바이트 (Bytes)`** (확장 헤더를 제외한 기본 헤더가 40 Bytes 고정적)

## 168. 데이터베이스 구축 - SQL 응용 (다중 테이블 INNER JOIN 및 GROUP BY 부서별 AVG 평균 급여 산출 후 ORDER BY DESC 내림차순 정렬)

### 문제
다음 `[EMP]` 및 `[DEPT]` 테이블을 참고하여 아래 SQL 쿼리문의 실행 결과를 쓰시오.

**[EMP]**
| emp_no | emp_nm | dept_id | salary |
| :---: | :---: | :---: | :---: |
| 101 | 홍길동 | 10001 | 3700 |
| 101 | 김철수 | 10002 | 4200 |
| 102 | 이만기 | 10003 | 5000 |
| 104 | 박지훈 | 10001 | 4800 |

**[DEPT]**
| dept_id | dept_name |
| :---: | :---: |
| 10001 | 총무팀 |
| 10002 | 전산팀 |
| 10003 | 회계팀 |

**[SQL]**
```sql
SELECT d.dept_name, AVG(e.salary) AS avg_salary
FROM EMP e
JOIN DEPT d ON e.dept_id = d.dept_id
GROUP BY d.dept_name
ORDER BY avg_salary DESC;
```

### 정답 및 풀이 결과
* **작성한 답**:
  `dept_name : 회계팀, 총무팀, 전산팀 / avg_salary : 5000, 4250, 4200` (JOIN 매핑 후 부서별 AVG 급여 계산 및 ORDER BY DESC 내림차순 정렬 순서를 정확히 계산하여 100% 정답 도출 성공)
* **모범 답안**:
  | dept_name | avg_salary |
  | :--- | :--- |
  | 회계팀 | 5000 |
  | 총무팀 | 4250 |
  | 전산팀 | 4200 |
  *(※ 표 형태 또는 컬럼별 값 연결 행 형태로 작성 시 모두 정답 인정)*
* **결과**: **정답 (100% 매칭)**

### 핵심 해설
이 문제는 RDB의 **내부 조인(INNER JOIN)** 과 **그룹화 집계 함수(`GROUP BY` & `AVG`)**, 그리고 **결과 정렬(`ORDER BY DESC`)** 의 실행 순서와 데이터 산출 메커니즘을 정밀하게 계산할 수 있는지 검증하는 빈출 쿼리 문제입니다.

* **단계별 쿼리 트레이싱 및 정밀 연산 과정**:
  1. **`JOIN DEPT d ON e.dept_id = d.dept_id` (조인 결합)**:
     * 10001 번 부서 (총무팀): 홍길동(3700), 박지훈(4800)
     * 10002 번 부서 (전산팀): 김철수(4200)
     * 10003 번 부서 (회계팀): 이만기(5000)
  2. **`GROUP BY d.dept_name` & `AVG(e.salary)` (부서별 평균 계산)**:
     * **회계팀**: $5000 / 1 =$ **`5000`**
     * **총무팀**: $(3700 + 4800) / 2 = 8500 / 2 =$ **`4250`**
     * **전산팀**: $4200 / 1 =$ **`4200`**
  3. **`ORDER BY avg_salary DESC` (평균 급여 내림차순 정렬)**:
     * 1위 (5000): **회계팀**
     * 2위 (4250): **총무팀**
     * 3위 (4200): **전산팀**
