## 43. Java - 비트 연산자 (NOT, XOR, AND) 및 산술 연산

### 문제
아래 자바 코드를 실행했을 때 출력되는 결과를 쓰시오. (단, 3라인 끝의 `；` 등 OCR 오타는 무시하고 올바른 자바 문법으로 간주하여 풀이한다.)
```java
public class Soojebi {
    public static void main(String[ ] args) {
        int a=15;
        int b=9;
        System.out.println(~a+(a^b)+(a & b));
    }
}
```

### 정답 및 풀이 결과
* **작성한 답**: `-1` (비트 연산자 원리 및 연산자 우선순위를 정확히 대입하여 풀이 성공)
* **모범 답안**: **`-1`**
* **결과**: **정답 (100% 매칭 및 계산 과정 완벽)**

### 핵심 해설
* **비트 연산자별 계산 상세 디버깅**:
  1. **`~a` (비트 NOT 연산)**:
     * 10진수 `15`를 8비트 이진수로 표현하면 `0000 1111`입니다.
     * 모든 비트를 반전시키는 비트 NOT(`~`) 연산을 수행하면 `1111 0000`이 됩니다. 
     * 컴퓨터의 2의 보수 표현법에 따라, 비트 NOT 연산의 결과는 `~x = -(x + 1)` 공식을 만족하므로 `~15 = -(15 + 1) = -16`이 됩니다.
  2. **`a ^ b` (비트 XOR 연산)**:
     * 두 비트가 서로 다르면 `1`, 같으면 `0`을 반환하는 연산입니다.
     * `15` (`0000 1111`) XOR `9` (`0000 1001`) = `0000 0110`
     * 이진수 `0110`을 10진수로 변환하면 **`6`**이 됩니다.
  3. **`a & b` (비트 AND 연산)**:
     * 두 비트가 모두 `1`일 때만 `1`, 하나라도 `0`이면 `0`을 반환하는 연산입니다.
     * `15` (`0000 1111`) AND `9` (`0000 1001`) = `0000 1001`
     * 이진수 `1001`을 10진수로 변환하면 **`9`**가 됩니다.
* **최종 연산 과정**:
  * 식: `~a + (a^b) + (a & b)`
  * 대입: `-16 + 6 + 9`
  * 결과: `-16 + 15 = -1`
  * 따라서 화면에 **`-1`**이 출력됩니다.

## 44. 데이터베이스 - 트랜잭션 병행 제어(Concurrency Control)의 로킹(Locking) 기법

### 문제
트랜잭션이 사용하는 데이터 항목에 대하여 잠금(Lock)을 설정한 트랜잭션이 해제(Unlock)할 때까지 독점적으로 사용할 수 있게 상호배제 기능을 제공하는 기법은 무엇인지 쓰시오.

### 정답 및 풀이 결과
* **작성한 답**: `locking` (영문 표기 매칭 및 병행 제어 주요 기법/로킹 단위 특징 분석 요청 포함)
* **모범 답안**: **`로킹 (Locking)`** (또는 `로킹 기법`)
* **결과**: **정답 (100% 매칭 및 개념 학습 완료)**

### 핵심 해설
* **로킹 (Locking) 기법**:
  * **정의**: 다중 사용자 환경에서 동시에 여러 트랜잭션이 실행될 때 데이터베이스의 일관성과 무결성을 지키기 위해, 특정 데이터 항목에 접근하기 전 잠금(Lock)을 획득하고 연산 완료 후 잠금을 해제(Unlock)하여 타 트랜잭션의 접근을 물리적으로 배제하는 기법입니다.
* **병행 제어의 4대 주요 기법 비교**:
  1. **로킹 (Locking)**:
     * **키워드**: "잠금(Lock)", "해제(Unlock)", "독점적 사용(상호배제)" (본 문제의 지문 내용)
  2. **타임스탬프 순서 (Timestamp Ordering)**:
     * **키워드**: "시스템이 부여한 고유 시간(Timestamp) 기준 순서 결정" (잠금(Lock)을 걸지 않음)
  3. **최적 병행 제어 (Optimistic / 낙관적 검증)**:
     * **키워드**: "충돌이 거의 없을 것이라 가정", "트랜잭션 종료 시 검증(Validation) 단계"
  4. **다중 버전 병행 제어 (MVCC / Multi-Version Concurrency Control)**:
     * **키워드**: "데이터의 다중 버전 유지", "읽기/쓰기 충돌 회피"
* **실기시험 단골: 로킹 단위(Locking Granularity)의 특징**:
  * **로킹 단위가 크면** (예: 데이터베이스, 테이블 단위):
    * 로크(Lock)의 개수가 **적어집니다**.
    * 병행 제어 기법이 **단순**하고 오버헤드가 적습니다.
    * 그러나 한 번에 차단되는 영역이 넓으므로 **병행성(공유 수준)이 낮아집니다** (병목 발생).
  * **로킹 단위가 작으면** (예: 속성, 레코드 단위):
    * 로크(Lock)의 개수가 **많아집니다**.
    * 병행 제어가 **복잡**해지고 오버헤드가 커집니다.
    * 하지만 상세 세부 단위만 잠그므로 **병행성(공유 수준)이 높아집니다**.
